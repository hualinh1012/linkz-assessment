## Purpose

This document defines how the codebase is structured. The platform follows **Clean Architecture**: business rules (entities) and application logic (use cases) are isolated from the delivery mechanism (Next.js Route Handlers), the frameworks, and the I/O (PostgreSQL/Prisma, Keycloak, AXS). Those external concerns sit behind **ports** (interfaces) and are implemented by **adapters** in the outer layer.

This is the structural foundation the rest of the General section plugs into — the [1.3. Database Schema](./1.3.%20Database%20Schema.md), [1.5. Authentication Setup](./1.5.%20Authentication%20Setup.md), and [1.6. Payment Integration](./1.6.%20Payment%20Integration.md) all describe **infrastructure adapters**, while the [2. Critical Flows](./2.%20Critical%20Flows.md) describe **use cases** orchestrating them.

---

## 1. Why Clean Architecture Here

The platform is explicitly designed to evolve from 3 fixed seats to a 100K-CCU system *without a rewrite* (see [SAD/1.1. Target Architecture](../SAD/1.1.%20Target%20Architecture.md)). That evolution changes **infrastructure** — adding a Redis cache, splitting reads onto Aurora replicas, making the payment callback event-driven — while the **business rules stay identical**.

Clean Architecture makes that possible: each of those changes is a new adapter behind a port (some ports, like the cache, don't exist yet and are introduced at that time). The domain and use-case layers never import Prisma, the AXS client, a cache SDK, or `next/*` — so they never change when the infrastructure does.

## 2. Layers & The Dependency Rule

```
        ┌─────────────────────────────────────────────────────┐
        │  Frameworks & Drivers (outermost)                    │
        │  Next.js App Router, Prisma, Redis SDK (later),      │
        │  AXS HTTP client, Keycloak/jose, Docker, AWS         │
        │   ┌─────────────────────────────────────────────┐    │
        │   │  Interface Adapters                          │    │
        │   │  Route Handlers (controllers), middleware,   │    │
        │   │  repositories, gateways, presenters/mappers  │    │
        │   │   ┌─────────────────────────────────────┐    │    │
        │   │   │  Application (use cases)             │    │    │
        │   │   │  ReserveSeat, ListSeats,             │    │    │
        │   │   │  ConfirmPayment, CancelReservation,  │    │    │
        │   │   │  depend only on PORT interfaces      │    │    │
        │   │   │   ┌─────────────────────────────┐    │    │    │
        │   │   │   │  Domain (innermost)         │    │    │    │
        │   │   │   │  Seat, Reservation,         │    │    │    │
        │   │   │   │  Payment, User entities;    │    │    │    │
        │   │   │   │  enums; invariants; errors  │    │    │    │
        │   │   │   └─────────────────────────────┘    │    │    │
        │   │   └─────────────────────────────────────┘    │    │
        │   └─────────────────────────────────────────────┘    │
        └─────────────────────────────────────────────────────┘

   Dependencies point INWARD only. Inner layers define interfaces;
   outer layers implement them. The domain depends on nothing.
```

| Layer | Responsibility | May import | Must NOT import |
|-------|----------------|------------|-----------------|
| **Domain** | Entities, enums, value objects, invariants, domain errors | Nothing (pure TypeScript) | Prisma, Next.js, any SDK |
| **Application (use cases)** | Orchestrate domain + ports to fulfil one flow | Domain, port interfaces | Concrete adapters, Next.js, Prisma |
| **Interface Adapters** | Translate HTTP ⇄ use cases; implement ports | Application, Domain, SDKs | — |
| **Frameworks & Drivers** | Next.js runtime, DB, cache, gateways, config | Everything | — |

## 4. Ports & Adapters

A **port** is an interface owned by the application layer, expressed in domain terms. An **adapter** is an outer-layer class that implements it against a real technology. Use cases receive ports via constructor injection and never know which adapter is behind them.

```typescript
// application/ports/payment-gateway.ts  (port)
export interface PaymentGateway {
  buildPaymentPageUrl(input: PaymentInit): Promise<string>;
  verifyWebhook(rawBody: string, signature: string): boolean;
  queryTransaction(ref: string): Promise<PaymentResult>;
  voidOrRefund(ref: string, amountCents: number): Promise<ReversalResult>;
}

// infrastructure/payment/axs-payment-gateway.ts  (adapter)
export class AxsPaymentGateway implements PaymentGateway { /* … */ }
```

The use case depends only on `PaymentGateway`; the AXS specifics live entirely in the adapter. If the gateway is ever replaced, only the adapter and `container.ts` change.

The same pattern applies to user provisioning:

```typescript
// application/ports/user-repository.ts  (port)
export interface UserRepository {
  upsertByKeycloakId(input: { keycloakId: string; email: string }): Promise<User>;
}

// infrastructure/persistence/prisma-user-repository.ts  (adapter)
export class PrismaUserRepository implements UserRepository { /* … */ }
```

The same pattern applies to every external dependency:

| Port | Adapter(s) | Spec |
|------|-----------|------|
| `SeatRepository` / `ReservationRepository` / `PaymentRepository` | Prisma (Postgres) | [1.3. Database Schema](./1.3.%20Database%20Schema.md) |
| `UserRepository` | `PrismaUserRepository` | [1.5. Authentication Setup §3.6](./1.5.%20Authentication%20Setup.md) |
| `PaymentGateway` | AXS Checkout | [1.6. Payment Integration](./1.6.%20Payment%20Integration.md) |
| `TokenVerifier` | Keycloak / JWKS (`jose`) | [1.5. Authentication Setup](./1.5.%20Authentication%20Setup.md) |
| `Clock` | System clock (injected for testable 15-min expiry) | [1.3 §1.5–§1.6](./1.3.%20Database%20Schema.md) |
| `CacheProvider` *(deferred)* | None today — `ListSeats` reads the repository directly. A Redis adapter is introduced behind this port only when volume justifies it | [SAD/2.3. ADR - Caching](../SAD/2.3.%20ADR%20-%20Caching.md) |

> **Caching is intentionally absent at this stage** (see [SAD/2.3. ADR - Caching](../SAD/2.3.%20ADR%20-%20Caching.md)). There is no cache port, adapter, or eviction logic in the current codebase; seat reads go straight to PostgreSQL. The row above records the seam where a cache will slot in later — adding it is purely additive.

## 5. Controllers (Route Handlers)

Next.js Route Handlers under `src/app/api/` are **controllers** — the thinnest possible layer. A handler:

1. Reads/validates the HTTP request (auth context comes from `middleware.ts`).
2. Calls a single use case with plain input.
3. Maps the use-case result — or a thrown domain error — to the HTTP response and `{ error, code }` shape defined in [1.4. API Specification §2.3](./1.4.%20API%20Specification.md).

No business logic, SQL, locking, signature verification, or cache handling lives in a Route Handler — those belong to use cases and adapters. This keeps the delivery mechanism replaceable (a future worker/queue consumer in the target architecture can drive the *same* use cases).

```typescript
// app/api/reservations/route.ts  (controller)
export async function POST(req: Request) {
  const { seatId } = await req.json();
  const userId = req.auth.sub;                       // set by middleware
  try {
    const result = await container.reserveSeat.execute({ userId, seatId });
    return Response.json(result, { status: 201 });
  } catch (e) {
    return toHttpError(e);                            // domain error → { error, code } + status
  }
}
```

## 6. Composition Root (Wiring / DI)

`src/container.ts` is the **only** place that constructs concrete adapters and injects them into use cases. It is the seam where environment configuration (see [1.7. Environment & Configuration](./1.7.%20Environment%20%26%20Configuration.md)) selects implementations.

```typescript
// src/container.ts  (composition root)
const seats     = new PrismaSeatRepository(prisma);
const reserves  = new PrismaReservationRepository(prisma);
const payments  = new PrismaPaymentRepository(prisma);
const gateway   = new AxsPaymentGateway(/* AXS_* env */);
const clock     = new SystemClock();

export const container = {
  listSeats:         new ListSeatsUseCase(seats),                                  // reads PostgreSQL directly — no cache
  reserveSeat:       new ReserveSeatUseCase(seats, reserves, payments, gateway, clock),
  confirmPayment:    new ConfirmPaymentUseCase(reserves, payments, seats, gateway),
  cancelReservation: new CancelReservationUseCase(reserves, payments, seats, gateway, clock),
};
// When a cache is introduced (ADR 2.3): construct a RedisCacheProvider here and pass it to ListSeats — nothing else changes.
```

Because construction is centralised, no inner-layer code ever references `process.env` or a concrete class.

## 8. Testability

The layering yields a clear test pyramid:

- **Domain & use-case tests** — pure unit tests with in-memory fake adapters (e.g. a fake `PaymentGateway`, in-memory repositories) and an injected fixed `Clock`. No DB, no network — fast and deterministic, including the 15-minute expiry logic.
- **Adapter (integration) tests** — exercise Prisma against a real Postgres (Docker Compose, see [1.7 §5.3](./1.7.%20Environment%20%26%20Configuration.md)) and AXS against its sandbox; verify the `SELECT FOR UPDATE` transaction and partial unique indexes from [1.3 §1.4–§1.5](./1.3.%20Database%20Schema.md).
- **End-to-end tests** — drive the Route Handlers / pages through the full container.

---

## 9. Rules of Thumb (for reviewers)

- A domain or use-case file importing `@prisma/client`, the AXS client, or `next/*` is an architecture violation — move that concern into an adapter behind a port.
- New external dependency → define a port first, implement the adapter second, wire it in `container.ts` last.
- Business rules and state transitions live in the domain/use case, **never** in a Route Handler.
- When the target architecture swaps a technology, expect to add an adapter and edit `container.ts` — and nothing else.
