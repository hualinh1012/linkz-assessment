## 4.1 AXS Checkout — Payment Link Flow

The integration uses the **AXS Payment Link** (redirect-based hosted checkout). The application never touches card data — the user is redirected to an AXS-hosted page for the payment step.

```
User                        Application                      AXS
 │                               │                            │
 │  POST /api/reservations       │                            │
 │──────────────────────────────►│                            │
 │                               │  Create PENDING reservation│
 │                               │  + INITIATED payment (DB)  │
 │                               │  Encrypt JWE payment link  │
 │  201 { redirectUrl }          │                            │
 │◄──────────────────────────────│                            │
 │                               │                            │
 │  Browser ──── redirectUrl ────────────────────────────────►│
 │                               │         AXS Payment Page   │
 │                               │  (cards / PayNow / wallets)│
 │                               │                            │
 │      User completes / declines / abandons payment          │
 │                               │                            │
 │◄── successUrl or failUrl ─────────────────────────────────-│
 │  (browser back on our site)   │                            │
 │                               │                            │
 │                               │◄── POST webhookUrl (JWE) ──│
 │                               │  Decrypt JWE               │
 │                               │  Idempotency check         │
 │                               │  Update DB state           │
 │                               │── 200 OK ─────────────────►│
```

**Key ordering**: AXS redirects the user to `successUrl` or `failUrl` first, then delivers the webhook server-to-server. The `successUrl`/`failUrl` pages are for user navigation only. **Never update DB state based on the browser redirect URL — wait for the webhook.**

## 4.2 Building the AXS Payment Link URL

The payment link is constructed by encrypting a parameter set using **JWE (JSON Web Encryption)** and appending it to the AXS checkout endpoint.

### Endpoint

```
GET https://checkout.svc.uat.pay.axs.com.sg/hpp/checkout/{merchantLinkId}?data={JWE}
```

- `merchantLinkId` — issued during AXS merchant onboarding (stored in `AXS_MERCHANT_LINK_ID`).
- `data` — JWE compact serialisation of the encrypted payload.

### JWE Header

| Field | Value |
|-------|-------|
| `alg` | `PBES2-HS512+A256KW` |
| `enc` | `A256GCM` |
| `p2s` | Random UUID per request (salt) |
| `p2c` | `1000` (fixed iteration count) |
| `kid` | `clientId` (merchant's AXS Client ID) |

Encrypted with `AXS_API_SECRET`.

### Payload Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `clientId` | string | **Yes** | AXS-issued merchant client ID |
| `amount` | integer | **Yes** | Amount in cents (e.g. `1500` = SGD 15.00); max 999 999 900 |
| `currency` | string | **Yes** | Always `"SGD"` |
| `merchantRef` | string | **Yes** | Our `paymentId` (UUID, max 40 chars) — correlates webhook → payment record |
| `webhookUrl` | string | No | `https://<host>/api/public/payment/callback` — overrides the portal-preset URL |
| `successUrl` | string | No | `https://<host>/reservation/confirmed?id=<reservationId>` |
| `failUrl` | string | No | `https://<host>/seats?error=payment_failed` |
| `scheme` | string | No | Comma-separated: `credit_card,paynow,alipay,wechatpay` (default: all) |

### TypeScript Implementation

```typescript
// src/infrastructure/payment/AxsPaymentGateway.ts

import * as jose from 'jose';
import { randomUUID } from 'crypto';

interface PaymentLinkParams {
  paymentId: string;       // our payment UUID → becomes merchantRef
  reservationId: string;   // used in successUrl
  amountCents: number;
  baseUrl: string;         // e.g. https://linkz.example.com
}

export async function buildPaymentLink(params: PaymentLinkParams): Promise<string> {
  const secret = new TextEncoder().encode(process.env.AXS_API_SECRET!);

  const jwe = await new jose.CompactEncrypt(
    new TextEncoder().encode(JSON.stringify({
      clientId:    process.env.AXS_CLIENT_ID!,
      amount:      params.amountCents,
      currency:    'SGD',
      merchantRef: params.paymentId,
      webhookUrl:  `${params.baseUrl}/api/public/payment/callback`,
      successUrl:  `${params.baseUrl}/reservation/confirmed?id=${params.reservationId}`,
      failUrl:     `${params.baseUrl}/seats?error=payment_failed`,
    }))
  )
    .setProtectedHeader({
      alg: 'PBES2-HS512+A256KW',
      enc: 'A256GCM',
      p2s: randomUUID(),
      p2c: 1000,
      kid: process.env.AXS_CLIENT_ID!,
    })
    .encrypt(secret);

  return `${process.env.AXS_CHECKOUT_URL}/hpp/checkout/${process.env.AXS_MERCHANT_LINK_ID}?data=${encodeURIComponent(jwe)}`;
}
```

## 4.3 Webhook Payload

AXS POSTs a **JWE-encrypted** body to the `webhookUrl` after every completed payment session. Decrypt it using the same `AXS_API_SECRET` before reading any fields.

### Decrypted fields

| Field | Type | Values / Notes |
|-------|------|----------------|
| `transactionRef` | string | AXS unique transaction ID |
| `merchantRef` | string | Our `paymentId` (passed in §4.2) |
| `status` | enum | `SUCCESS` \| `DECLINED` \| `EXPIRED` |
| `amount` | string | Dollar format, e.g. `"15.00"` |
| `currency` | string | `"SGD"` |
| `responseCode` | string | `"00"` = success; other codes = decline / error |
| `paymentScheme` | enum | `VISA` \| `MASTERCARD` \| `PAYNOW` \| `ALIPAY` \| `WECHATPAY` |
| `paymentDateTime` | string | ISO 8601 timestamp |

### TypeScript type

```typescript
interface AXSWebhookPayload {
  transactionRef: string;
  merchantRef: string;       // = our paymentId
  status: 'SUCCESS' | 'DECLINED' | 'EXPIRED';
  amount: string;            // dollar string e.g. "15.00"
  currency: string;
  responseCode: string;
  paymentScheme: string;
  paymentDateTime: string;   // ISO 8601
}
```

## 4.4 Webhook JWE Decryption & Validation

Every incoming webhook body is a JWE compact serialisation. Decrypt it with `AXS_API_SECRET` before any database access.

```typescript
// src/infrastructure/payment/AxsPaymentGateway.ts

import * as jose from 'jose';

export async function decryptAXSWebhook(rawBody: string): Promise<AXSWebhookPayload> {
  const secret = new TextEncoder().encode(process.env.AXS_API_SECRET!);

  const { plaintext } = await jose.compactDecrypt(rawBody.trim(), secret);
  const payload = JSON.parse(new TextDecoder().decode(plaintext)) as AXSWebhookPayload;

  if (!payload.transactionRef || !payload.merchantRef || !payload.status) {
    throw new Error('Invalid AXS webhook payload: missing required fields');
  }

  return payload;
}
```

Requests where `compactDecrypt` throws are rejected with `401 Unauthorized` before any further processing. Always pass the **raw** request body string, not a parsed object.

## 4.5 Idempotent Webhook Processing

AXS may deliver the same webhook more than once under network retry conditions. The handler is idempotent:

```typescript
const payment = await prisma.payment.findUnique({ where: { id: payload.merchantRef } });

if (payment && ['SUCCESS', 'FAILED', 'EXPIRED', 'VOIDED', 'REFUNDED'].includes(payment.status)) {
  return Response.json({ received: true }); // already terminal — no re-apply
}
```

## 4.6 Void vs Refund

When a user cancels a **CONFIRMED** reservation via `DELETE /api/reservations/:id`, the application reverses the AXS payment. The approach is **void-first, refund-on-failure**: attempt void; if AXS rejects it (payment already settled), fall back to a full refund.

```typescript
// src/infrastructure/payment/AxsPaymentGateway.ts

export async function reversePayment(
  axsTransactionRef: string,
  paymentId: string,
  amountCents: number,
): Promise<'VOIDED' | 'REFUNDED'> {
  const headers = {
    Authorization: `Bearer ${process.env.AXS_API_KEY}`,
    'Content-Type': 'application/json',
  };

  const voidRes = await fetch(
    `${process.env.AXS_API_BASE_URL}/transactions/${axsTransactionRef}/void`,
    { method: 'POST', headers, body: JSON.stringify({ merchantRef: paymentId }) },
  );
  if (voidRes.ok) return 'VOIDED';

  const refundRes = await fetch(
    `${process.env.AXS_API_BASE_URL}/transactions/${axsTransactionRef}/refund`,
    {
      method: 'POST',
      headers,
      body: JSON.stringify({
        merchantRef: paymentId,
        amount: (amountCents / 100).toFixed(2),   // dollar string
      }),
    },
  );
  if (!refundRes.ok) throw new Error('AXS refund failed');
  return 'REFUNDED';
}
```

If both void and refund fail, the reservation is left `CONFIRMED` and a `502 PAYMENT_REVERSAL_FAILED` is returned — no partial state is committed.

## 4.7 PENDING Expiry During Active Payment

The 15-minute `PENDING` expiry aligns with the AXS maximum payment-page session duration:

- A user cannot be on a valid AXS payment page after their reservation has expired.
- When the AXS session times out, AXS delivers an `EXPIRED` webhook and redirects the user to `failUrl`.
- The expiry cleanup job (see [2.5](./2.5.%20Expired%20Payment%20Clean%20Up.md)) only releases seats where `expires_at < now()`, the same window the AXS session would also be expired.

## 4.8 AXS Sandbox vs Production

| Environment | `AXS_CHECKOUT_URL` | `AXS_API_BASE_URL` |
|-------------|-------------------|-------------------|
| Development / Staging | `https://checkout.svc.uat.pay.axs.com.sg` | `https://api.svc.uat.axsasia.com` |
| Production | `https://checkout.svc.pay.axs.com.sg` | `https://api.svc.axsasia.com` |

Credentials (`AXS_API_SECRET`, `AXS_CLIENT_ID`, `AXS_MERCHANT_LINK_ID`, `AXS_API_KEY`) are stored in AWS Secrets Manager and injected as environment variables. See [1.7. Environment & Configuration](./1.7.%20Environment%20%26%20Configuration.md).

Local development requires a tunnelling tool (e.g. ngrok) to expose `POST /api/public/payment/callback` publicly so the AXS sandbox can deliver webhooks.
