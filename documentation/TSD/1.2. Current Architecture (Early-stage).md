# 1.2. Current Architecture (Early-stage)

**Status:** CURRENT  
**Date:** 2026-06-17  
**Authors:** HuaHoangLinh

## Purpose

This document describes the **architecture as it is actually built and deployed today** — the lean, early-stage version that serves the 3-seat platform. It is the concrete starting point that the [SAD/1.1. Target Architecture](../SAD/1.1.%20Target%20Architecture.md) (the 100K-CCU vision) evolves *from*, one trigger at a time.

The guiding principle here is **lean**: one deployable unit, one database, a hosted payment page, and a managed identity provider — **no cache**, no queues, no replicas, no CDN. Everything that the target architecture adds later is deliberately *absent* now and listed in §4.

## 3. Runtime Characteristics

- **Reservation safety** comes from PostgreSQL `SELECT FOR UPDATE` + partial unique indexes (see [1.3 §1.4–§1.5](./1.3.%20Database%20Schema.md)) — sufficient and simplest at 3 seats / low concurrency.
- **No cache.** Seat availability is read straight from PostgreSQL — a sub-millisecond indexed query over 3 rows. There is no cache to keep consistent and no invalidation logic; the DB is always the source of truth. A shared Redis cache is the first thing added at scale (see §4 and [SAD/2.3. ADR - Caching](../SAD/2.3.%20ADR%20-%20Caching.md)).
- **Payment is synchronous.** The AXS webhook is handled inline in the `/api/public/payment/callback` Route Handler; no queue or event bus.
- **Stateless app tasks.** No server-side session store in the app — identity is a Keycloak-issued JWT verified per request via JWKS. Tasks can be replaced freely.

## 4. Deliberately Deferred (the gap to Target)

These are intentionally **not** built yet; each is introduced only when a measurable trigger fires (full table in [SAD/1.1. Target Architecture](../SAD/1.1.%20Target%20Architecture.md)):

| Deferred | Replaced by, at scale | Trigger |
|----------|----------------------|---------|
| No cache (reads hit PostgreSQL) | ElastiCache Redis (shared) | Read load rises, or fleet needs shared state |
| Single RDS | RDS Proxy + Aurora read replicas | Connection limit / read latency |
| 2 fixed tasks | Auto-scaled ECS fleet | Sustained load |
| `SELECT FOR UPDATE` | SQS FIFO + reservation worker | DB lock contention |
| Synchronous webhook | EventBridge consumer | Payment callback spikes |
| ALB only | CloudFront + WAF + Shield | DDoS / bot traffic |

Because of the [Clean Architecture](./1.1.%20Project%20Architecture.md) boundaries, each of these is an infrastructure/adapter swap — none touches the domain or use-case code.
