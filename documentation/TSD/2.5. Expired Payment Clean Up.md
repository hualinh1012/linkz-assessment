# 2.5. Expired Payment Clean Up

**Status:** DRAFT  
**Date:** 2026-06-17  
**Authors:** HuaHoangLinh

---

## Purpose

This document describes the background process that detects **unfinished payments** — reservations left in `PENDING` because the user never completed the AXS Payment Page — and brings the system back to a clean state: the orphaned payment is marked terminal (`EXPIRED`), the reservation is `CANCELLED`, and the seat is released to `AVAILABLE`.

Without this process a seat could stay held forever after a user abandons checkout. It is the safety net that guarantees the 15-minute hold from [2.3. Reserve Seat](./2.3.%20Reserve%20Seat.md) is always honoured even when no webhook or return redirect ever arrives. The DB-level mechanics are specified in [1.3. Database Schema §1.6](./1.3.%20Database%20Schema.md).

---

## 1. What Needs Cleaning Up

After [2.3. Reserve Seat](./2.3.%20Reserve%20Seat.md) commits, the system is in this state for up to 15 minutes:

```
reservations.status = PENDING   (expires_at = created_at + 15 min)
seats.status        = PENDING
payments.status     = INITIATED
```

If the user pays or cancels, [2.4. Payment & Callback](./2.4.%20Payment%20%26%20Callback.md) drives the transition. But if the user simply **walks away** — closes the tab, loses connectivity, the AXS session lapses — none of those callbacks fire, and the three rows above are stranded. The cleanup job resolves exactly this case.

---

## 2. Trigger

> **⚠ Development / testing approach only** — see below for the real deployment pattern.

### Current implementation — `instrumentation.ts` in-process timer

`src/instrumentation.ts` uses Next.js's stable [Instrumentation API](https://nextjs.org/docs/app/building-your-application/optimizing/instrumentation) to register a `setInterval` at server boot. Every **5 minutes** it calls `ExpireReservationsUseCase` directly (no HTTP round-trip):

```
server start
  └─ instrumentation.ts:register()
       └─ run() immediately (catches anything expired while server was down)
       └─ setInterval(run, 5 min)
                └─ ExpireReservationsUseCase → PrismaReservationRepository.expireOverdue()
```

This is convenient for local development and single-instance staging — everything runs in one process with no extra infrastructure. **It is not suitable for production** because:
- Each horizontally-scaled instance runs its own timer (redundant DB load, though the CTE is idempotent so correctness is unaffected).
- If the app process is down, no cleanup runs until it restarts.

### Cron route — `GET /api/cron/cleanup-expired`

A manually-triggerable endpoint is also provided at `src/app/api/cron/cleanup-expired/route.ts`. It is protected by the `x-cron-secret` header (returns 404 if the secret is absent or wrong). This is useful for:
- **Ad-hoc testing** without waiting 5 minutes for the timer.
- **Real deployment**: point an external scheduler at this endpoint instead of relying on the in-process timer.

```bash
curl -H "x-cron-secret: $CRON_SECRET" https://your-host/api/cron/cleanup-expired
# → { "released": 0 }
```

### Real deployment — Option B or C (required for production)

The in-process timer **must be replaced** with an external scheduler before going to production. Two approaches:

**Option B — Docker Compose cron service** (self-hosted / staging)

Add a lightweight service to `docker-compose.yml` that calls the cron route at the desired interval:

```yaml
cron:
  image: alpine/curl
  restart: unless-stopped
  entrypoint: >
    sh -c "while true; do
      curl -sf -H 'x-cron-secret: $$CRON_SECRET'
           http://app:3000/api/cron/cleanup-expired;
      sleep 300;
    done"
  environment:
    CRON_SECRET: ${CRON_SECRET}
  depends_on:
    - app
```

**Option C — AWS EventBridge scheduled rule** (production on AWS)

Create an EventBridge rule on a `rate(5 minutes)` schedule targeting either:
- A Lambda function that calls the cron route (HTTP), or
- An ECS task that runs the cleanup SQL directly against the DB.

This decouples the cleanup from the app fleet and is the target architecture (see [SAD/1.1. Target Architecture](../SAD/1.1.%20Target%20Architecture.md)).

### Cadence

The 5-minute cadence is well within the 15-minute `PENDING` expiry window — a stranded seat is released within ~5 minutes of its `expires_at`. The 1-minute cadence from [1.3 §1.6](./1.3.%20Database%20Schema.md) represents the target production precision; 5 minutes is the pragmatic default for this stage.

## 3. Detection Criteria

A payment is considered **unfinished / expired** when:

```
payments.status      = 'INITIATED'                 -- never reached a terminal state
AND reservations.status = 'PENDING'                -- hold never confirmed/cancelled
AND reservations.expires_at < now()                -- the 15-minute window has passed
```

`INITIATED` is the only non-terminal payment state, so it is the sole status the job touches. Payments already `SUCCESS | FAILED | EXPIRED | VOIDED | REFUNDED` are terminal and ignored.

---

## 4. Process

The cleanup runs as a single transaction so the three rows transition atomically (full SQL in [1.3 §1.6](./1.3.%20Database%20Schema.md)):

```
Cleanup job (every minute)
   │
   ▼
BEGIN
   │  1. Mark orphaned payments:
   │     payments.status: INITIATED → EXPIRED, completed_at = now()
   │     (where reservation is PENDING and expires_at < now())
   │
   │  2. Cancel the expired reservations:
   │     reservations.status: PENDING → CANCELLED
   │
   │  3. Release their seats:
   │     seats.status: PENDING → AVAILABLE
COMMIT
   │
   ▼
Next GET /api/seats reflects the freed seats (read directly from PostgreSQL — no cache)
```

Payments are transitioned **before** the reservations are cancelled, so the "still INITIATED + reservation still PENDING" predicate cleanly identifies the orphans (step 2 would otherwise erase the link the predicate relies on).

---

## 5. State Transitions

| Entity | Before | After |
|--------|--------|-------|
| `payments.status` | `INITIATED` | `EXPIRED` (`completed_at` set) |
| `reservations.status` | `PENDING` | `CANCELLED` |
| `seats.status` | `PENDING` | `AVAILABLE` |

(No cache at this stage — the next `GET /api/seats` shows `AVAILABLE` directly from PostgreSQL.)

---

## 6. `EXPIRED` vs `FAILED`

Both are terminal, but they mean different things and are recorded distinctly for debugging and metrics (see the `PaymentStatus` enum in [1.3 §1.2](./1.3.%20Database%20Schema.md)):

| Status | Meaning | Set by |
|--------|---------|--------|
| `EXPIRED` | User never completed the payment within 15 minutes (abandoned / timed out) | This cleanup job (§4) |
| `FAILED` | AXS actively declined / the payment attempt failed | Payment callback — [2.4 §3](./2.4.%20Payment%20%26%20Callback.md) |

Keeping them separate lets us distinguish *abandonment* from *payment failure* in reporting without inspecting timestamps.

---

## 7. Interaction With a Late Webhook

The 15-minute hold is aligned with the AXS Checkout maximum payment-page session (see [1.6 §4.8](./1.6.%20Payment%20Integration.md)), so a user cannot still be on a valid AXS page once their reservation has expired. Consequently:

- A genuine `SUCCESS` should never arrive after expiry.
- If a late webhook does arrive for an already-`EXPIRED` payment, the idempotency guard treats `EXPIRED` as terminal and ignores it (see [1.6 §4.5](./1.6.%20Payment%20Integration.md)) — the seat, already released, is not silently re-reserved. Such an event (paid-after-expiry) should be logged for manual reconciliation/refund, as it indicates an AXS session that outlived its expected window.

---

## 8. Edge Cases

| Scenario | Outcome |
|----------|---------|
| User completes payment at 14:59 of the window | Webhook confirms before the job runs; payment is `SUCCESS`, not picked up by the `INITIATED` predicate |
| Job runs while a reservation is exactly at `expires_at` | `expires_at < now()` is strict — a reservation is only cleaned once `now()` has passed it |
| Job runs twice (overlapping schedules) | Idempotent — the second run finds no `INITIATED` + `PENDING` + expired rows to change |
| Reservation already `CANCELLED` by the callback | Not matched (predicate requires `PENDING`); payment already terminal |
| Cleanup transaction fails midway | Rolled back; the rows remain `PENDING`/`INITIATED` and are retried on the next minute's run |
