## Purpose

This document describes the runtime authentication flow — how a user logs in via Keycloak, how the 90-day session is maintained, how the Next.js application verifies tokens on every protected request, and how a local user record is provisioned. It is the dynamic counterpart to the static config in [1.5. Authentication Setup](./1.5.%20Authentication%20Setup.md).

---

## 1. Login Flow (OIDC Authorization Code + PKCE)

The handshake is implemented by **Auth.js (NextAuth v5)** with the Keycloak provider — Auth.js performs the OIDC **Authorization Code flow with PKCE** against the self-hosted Keycloak `linkz` realm / `linkz-web` confidential client, exchanges the code, and writes the encrypted session cookie. The app does not hand-roll this flow. The `/api/auth/*` routes are served by Auth.js (`app/api/auth/[...nextauth]/route.ts`).

```
Browser                 Next.js App              Keycloak
   │                        │                       │
   │  Click "Login"         │                       │
   │───────────────────────►│                       │
   │                        │  Build authorize URL  │
   │                        │  (PKCE S256 challenge) │
   │  302 → Keycloak /authorize ───────────────────►│
   │◄───────────────────────────────────────────────│
   │                        │                       │
   │  Login UI (Keycloakify React theme)            │
   │  enter email + password                        │
   │───────────────────────────────────────────────►│
   │                        │      verify creds     │
   │◄─── 302 → /api/auth/callback?code=… ───────────│
   │                        │                       │
   │  GET /api/auth/callback?code=…                  │
   │───────────────────────►│                       │
   │                        │  exchange code+verifier│
   │                        │──────────────────────►│
   │                        │◄── access + refresh ──│
   │                        │   + id token          │
   │                        │  set HTTP-only cookie  │
   │◄─── 302 → / (logged in)│                       │
```

**Steps**

1. The user clicks **Login**. The app generates a PKCE `code_verifier` / `code_challenge` (`S256`) and redirects the browser to Keycloak's `/protocol/openid-connect/auth` endpoint with `client_id=linkz-web`, `scope=openid profile email offline_access`, and the challenge.
2. Keycloak serves the **Keycloakify** login page (custom React + Tailwind theme — see [1.5 §3.7](./1.5.%20Authentication%20Setup.md)). The user authenticates with email + password (social/OTP providers are additive, per [1.5 §3.8](./1.5.%20Authentication%20Setup.md)).
3. Keycloak redirects back to a valid redirect URI (`https://linkz.example.com/*`) with an authorization `code`.
4. The app exchanges the `code` + `code_verifier` for an **access token**, **refresh token**, and **id token** at Keycloak's token endpoint.
5. The **refresh token is stored in an HTTP-only, Secure, SameSite cookie** — never exposed to client-side JavaScript. The short-lived access token is used for API calls.

## 2. Session Lifetime (90 days)

After login, the user's session is an **encrypted, HTTP-only cookie** issued by Auth.js (JWT strategy — no session table). Subsequent requests carry the cookie automatically; the app does **not** call Keycloak or exchange a token per request.

| Layer | Setting | Role |
|-------|---------|------|
| Auth.js session cookie | `session.maxAge = 7776000s` (90d) | The de-facto session for our own API — validated locally with `AUTH_SECRET` |
| Keycloak SSO Session | `SSO Session Max` / `Idle` = 7776000s (90d) | Authority for the OIDC login; checked at sign-in, not per request |

```
User signs in (OIDC + PKCE via Auth.js)
        │
        ▼
Encrypted session cookie set (maxAge 90 days)
        │
        ▼
Each request: cookie decrypted + verified locally (no Keycloak / JWKS call)
        │
        ▼
User stays logged in up to 90 days without re-authenticating
```

There is no per-request access-token refresh: because our API is self-contained (it never calls a Keycloak-protected downstream resource as the user), only the `sub`/email are kept in the session — no Keycloak access/refresh token is stored or refreshed.

### Caveats (track these in production — see [SAD/1.1. Target Architecture](../SAD/1.1.%20Target%20Architecture.md))

- **Session revocation** is handled via **Keycloak Back-Channel Logout**: when a user changes their password, an admin revokes their session, or a user is disabled, Keycloak POSTs a signed Logout Token (JWS) to `POST /api/auth/backchannel-logout`. The endpoint verifies the token via JWKS and upserts the user's `sub` into the `revoked_sessions` table (see [1.3 §1.7](./1.3.%20Database%20Schema.md)). The middleware checks this denylist on every protected request — a matched `sub` returns `401` immediately. At scale the denylist moves to the Redis store already in the target architecture; no middleware logic change is required.
- **`AUTH_SECRET` rotation** must be coordinated across the fleet; Auth.js supports a secret *array* for zero-downtime rotation. Rotating invalidates existing sessions (see above).
- **Rolling vs absolute.** Auth.js extends the cookie expiry on activity, so a continuously-active user can exceed a strict 90-day *absolute* cap. If an absolute cap is required, disable rolling (`session.updateAge`) or enforce an absolute max in the `jwt` callback.
- **Profile/role staleness.** `sub`/email are snapshotted at sign-in; Keycloak-side changes (email, roles) are not reflected until the next login. Matters only if RBAC is added.

## 3. Per-Request Session Verification

Every request to a protected `/api/*` route passes through `src/middleware.ts` (Edge) before reaching the Route Handler (see [1.4. API Specification §2.6](./1.4.%20API%20Specification.md)).

```
Request → src/middleware.ts  (Edge Auth.js from auth.config — no DB)
   │
   ├─ Public path? (/api/public/*  /api/auth/*) ─► pass through
   │
   ├─ Read Auth.js session cookie (decrypted with AUTH_SECRET)
   │
   ├─ session present ─► set x-user-id (= sub) + x-user-email → Route Handler
   └─ no session      ─► 401 UNAUTHORIZED (no DB access)
```

The session cookie is validated with `AUTH_SECRET`; no JWKS call is made per request (Auth.js verified the Keycloak ID token against JWKS during the handshake). The authenticated user identity (`sub`) reaches handlers via the `x-user-id` header. See [1.5 §3.5](./1.5.%20Authentication%20Setup.md).

## 4. User Provisioning on First Sign-In

The `users` table holds only a local profile reference; Keycloak is the source of truth for identity. On first sign-in, the Auth.js `signIn` event (Node runtime) upserts a local record keyed by `keycloak_id` (the `sub` claim) via the `ProvisionUser` use case — see [1.5 §3.6](./1.5.%20Authentication%20Setup.md).

```
Auth.js sign-in (Node) → events.signIn
   │
   ▼
ProvisionUser.execute({ keycloakId: sub, email })
   │
   ▼
UserRepository.upsertByKeycloakId  → prisma.user.upsert({ where: { keycloakId }, … })
```

The upsert is idempotent, so concurrent first sign-ins are safe.

---

## 5. Logout

1. `signOut()` clears the Auth.js session cookie.
2. Optionally, the browser is redirected to Keycloak's end-session endpoint (`Valid Post Logout Redirect URIs = https://linkz.example.com/`) to also terminate the Keycloak SSO session. Note: clearing the local cookie ends *this app's* session; without the Keycloak end-session call the user may be silently re-logged-in on next sign-in if the Keycloak SSO session is still alive.

## 6. Error & Edge Cases

| Scenario | Outcome |
|----------|---------|
| No / invalid session cookie on a protected route | `401 UNAUTHORIZED` at middleware, before any handler logic |
| Session cookie expired (>90 days) | Treated as no session → `401`; user signs in again |
| Tampered or wrong-`AUTH_SECRET` cookie | Decryption fails → treated as no session → `401` |
| Keycloak unreachable during login | Sign-in fails; user remains unauthenticated and can retry |
| Need to force-revoke a session before expiry | Not possible with stateless JWT sessions — see the Revocation caveat in §2 |
| First-sign-in upsert race (two concurrent sign-ins) | `keycloak_id` unique constraint + `upsert` makes provisioning idempotent |
