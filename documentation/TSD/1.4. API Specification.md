## Purpose

This document defines the complete HTTP API contract for the Linkz Seat Reservation Platform. All routes are Next.js 14 Route Handlers under `src/app/api/`. Authentication is enforced via JWT middleware on all non-public endpoints.

| Environment | Base URL |
|-------------|----------|
| Development | `http://localhost:3000` |
| Staging | `https://staging.linkz.example.com` |
| Production | `https://linkz.example.com` |

## 2.2 Authentication

All endpoints under `/api/*` except `/api/public/*` require an authenticated session. Authentication is handled by **Auth.js (NextAuth v5)** with the Keycloak OIDC provider: after login, the session is held in an encrypted, HTTP-only cookie (JWT strategy — no session table). Browser requests are same-origin, so the cookie is sent automatically.

`src/middleware.ts` reads the session on every protected request, returns `401 Unauthorized` if absent/invalid, and forwards the identity to Route Handlers via an `x-user-id` header (the Keycloak `sub`). The `userId` used throughout the request is that header value.

**Route layout convention**
- `/api/public/*` — no JWT required; each route self-guards with its own mechanism (JWE decryption, cron secret, test token, etc.)
- `/api/*` (everything else) — JWT session required, enforced by `src/middleware.ts`

## 2.3 Common Error Response Shape

```typescript
{
  error: string;    // Human-readable message
  code: string;     // Machine-readable code for client handling
}
```

Common error codes:

| `code` | HTTP Status | Meaning |
|--------|-------------|---------|
| `UNAUTHORIZED` | 401 | Missing or invalid JWT |
| `FORBIDDEN` | 403 | Authenticated but not permitted |
| `NOT_FOUND` | 404 | Resource does not exist |
| `CONFLICT` | 409 | Business rule violation (seat taken, duplicate reservation) |
| `UNPROCESSABLE` | 422 | Request shape valid but content invalid |
| `INTERNAL` | 500 | Unexpected server error |

## 2.4 Endpoints

---

### GET /api/public/seats

Returns the current availability status of all seats. No authentication required.

**Request**

```
GET /api/public/seats
```

**Response — 200 OK**

```typescript
{
  seats: Array<{
    id: number;
    label: string;
    status: 'AVAILABLE' | 'PENDING' | 'RESERVED';
  }>;
}
```

**Example**

```json
{
  "seats": [
    { "id": 1, "label": "Seat A", "status": "AVAILABLE" },
    { "id": 2, "label": "Seat B", "status": "PENDING" },
    { "id": 3, "label": "Seat C", "status": "RESERVED" }
  ]
}
```

**Notes**
- Response is read directly from PostgreSQL — there is no cache at this stage (see [SAD/2.3. ADR - Caching](../SAD/2.3.%20ADR%20-%20Caching.md)).
- Lives under `/api/public/` — no JWT required. Suitable for SSR on the public seat listing page.

---

### POST /api/reservations

Creates a PENDING reservation for the authenticated user on the requested seat, then initiates an AXS payment session. Returns the AXS Payment Page redirect URL.

**Request**

```
POST /api/reservations
Authorization: Bearer <access_token>
Content-Type: application/json
```

```typescript
{
  seatId: number;    // 1, 2, or 3
}
```

**Response — 201 Created**

```typescript
{
  reservation: {
    id: string;             // UUID
    seatId: number;
    status: 'PENDING';
    expiresAt: string;      // ISO 8601 — 15 minutes from now
  };
  payment: {
    id: string;             // UUID
    status: 'INITIATED';
  };
  redirectUrl: string;      // Signed AXS Payment Page URL — redirect client here
}
```

**Errors**

| Scenario | HTTP | `code` |
|----------|------|--------|
| Seat does not exist | 404 | `NOT_FOUND` |
| Seat is not AVAILABLE | 409 | `CONFLICT` |
| User already has active reservation | 409 | `CONFLICT` |
| AXS payment initiation failed | 502 | `PAYMENT_INITIATION_FAILED` |

**Notes**
- Runs inside a `SELECT FOR UPDATE` transaction — locks the seat row before writing.
- Seat status is set to `PENDING`; `expires_at` is set to `now() + 15 minutes`.
- The AXS Payment Page URL includes the `reservation_id` and a signed return URL.

---

### POST /api/public/payment/callback

AXS server-to-server webhook. AXS calls this URL after every completed payment session (SUCCESS, DECLINED, or EXPIRED). The browser never calls this endpoint — AXS redirects the user to `successUrl` or `failUrl` separately.

**Request**

```
POST /api/public/payment/callback
Content-Type: application/octet-stream   (JWE compact serialisation)
```

The request body is a **JWE compact serialisation** encrypted with `AXS_API_SECRET` (`PBES2-HS512+A256KW` / `A256GCM`). The decrypted payload:

```typescript
{
  transactionRef: string;
  merchantRef: string;      // = our paymentId
  status: 'SUCCESS' | 'DECLINED' | 'EXPIRED';
  amount: string;           // dollar string e.g. "15.00"
  currency: string;
  responseCode: string;
  paymentScheme: string;
  paymentDateTime: string;  // ISO 8601
}
```

**Response — 200 OK**

```typescript
{ received: true }
```

**State transitions on `SUCCESS`**

```
payments.status:              INITIATED → SUCCESS (axs_transaction_ref + completed_at set)
reservations.status:          PENDING   → CONFIRMED
seats.status:                 PENDING   → RESERVED
```

**State transitions on `DECLINED`**

```
payments.status:       INITIATED → FAILED (completed_at set)
reservations.status:   PENDING   → CANCELLED
seats.status:          PENDING   → AVAILABLE
```

**State transitions on `EXPIRED`**

```
payments.status:       INITIATED → EXPIRED (completed_at set)
reservations.status:   PENDING   → CANCELLED
seats.status:          PENDING   → AVAILABLE
```

> `FAILED` (gateway decline) and `EXPIRED` (user abandonment / timeout) are kept distinct — see [1.3 §1.3](./1.3.%20Database%20Schema.md) and [2.5 §6](./2.5.%20Expired%20Payment%20Clean%20Up.md).

**Security**
- Authentication is JWE decryption. Any forged or tampered body fails `compactDecrypt` and is rejected with `401 Unauthorized` before any DB access.
- Idempotent — payments already in a terminal state are acknowledged without re-applying state transitions.

### DELETE /api/reservations/:id

Cancels a CONFIRMED reservation, initiates an AXS void or refund, and releases the seat.

**Request**

```
DELETE /api/reservations/:id
Authorization: Bearer <access_token>
```

**Response — 200 OK**

```typescript
{
  reservation: {
    id: string;
    status: 'CANCELLED';
  };
  payment: {
    id: string;
    status: 'VOIDED' | 'REFUNDED';
  };
  seat: {
    id: number;
    status: 'AVAILABLE';
  };
}
```

**Errors**

| Scenario | HTTP | `code` |
|----------|------|--------|
| Reservation not found | 404 | `NOT_FOUND` |
| Reservation belongs to another user | 403 | `FORBIDDEN` |
| Reservation is not CONFIRMED | 409 | `CONFLICT` |
| AXS void/refund API failed | 502 | `PAYMENT_REVERSAL_FAILED` |

**Void vs Refund logic**

| Condition | AXS Action |
|-----------|------------|
| Payment settled same day (before AXS settlement cutoff) | `void` API call |
| Payment settled in prior period | `refund` API call |

The application checks the `payments.completed_at` timestamp against the AXS settlement window to determine which action to call. If neither succeeds, the reservation is left CONFIRMED and the error is surfaced to the user.

**State transitions on success**

```
reservations.status: CONFIRMED → CANCELLED
payments.status: SUCCESS → VOIDED | REFUNDED
seats.status: RESERVED → AVAILABLE
```

The released seat is reflected on the next read directly from PostgreSQL (no cache at this stage).

## 2.5 Route Summary

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/api/public/seats` | Public | List all seat statuses |
| `POST` | `/api/reservations` | JWT required | Reserve a seat + initiate payment |
| `GET` | `/api/reservations/:id` | JWT required | Poll reservation + payment status |
| `POST` | `/api/public/payment/callback` | JWE (self-guarded) | AXS server-to-server webhook |
| `DELETE` | `/api/reservations/:id` | JWT required | Cancel reservation + void/refund |
| `GET` | `/api/public/cron/cleanup-expired` | Cron secret (self-guarded) | Manual trigger for expiry cleanup job |
| `POST` | `/api/public/test/reset-seats` | Test token (self-guarded) | Dev/test backdoor — reset all seats |

## 2.6 Middleware

All routes under `/api/*` except `/api/public/*` and `/api/auth/*` pass through `src/middleware.ts` (Edge runtime) before reaching the Route Handler. The middleware builds an Auth.js instance from the edge-safe `auth.config.ts` (no DB import).

**Middleware responsibilities**

1. Bypass public API paths (single prefix check — see below).
2. Read the Auth.js session cookie (decrypted with `AUTH_SECRET`).
3. Return `401 { error, code: "UNAUTHORIZED" }` immediately if there is no valid session.
4. Forward identity to the handler via request headers `x-user-id` (= Keycloak `sub`) and `x-user-email`.

**Route layout**

```
/api/public/*   — no JWT; each route self-guards with its own mechanism:
                    /api/public/payment/callback  → JWE decryption (401 on failure)
                    /api/public/cron/*            → x-cron-secret header
                    /api/public/test/*            → x-test-token + non-production guard (404 if absent)
/api/auth/*     — Auth.js OIDC handshake; bypassed by middleware
/api/**         — JWT session required; middleware enforces
```

This single-prefix rule (`pathname.startsWith("/api/public/")`) means new public routes can never be accidentally left unguarded by the middleware — they must be placed under `/api/public/` and carry their own auth mechanism.

Pages are not guarded here; the public seat page is open and protected pages manage their own redirect via the `auth()` helper.

## 2.7 Rate Limiting

Rate limiting state is held **in-process per instance** at this stage (best-effort across the 2 ECS tasks — acceptable at current volume, since there is no shared cache). It moves to a shared store (ElastiCache Redis) when caching is introduced — see [SAD/2.3. ADR - Caching](../SAD/2.3.%20ADR%20-%20Caching.md) and [1.1. Target Architecture](../SAD/1.1.%20Target%20Architecture.md).

| Endpoint | Limit |
|----------|-------|
| `POST /api/reservations` | 5 requests / minute per user |
| `DELETE /api/reservations/:id` | 10 requests / minute per user |
| `GET /api/public/seats` | 60 requests / minute per IP |

Requests exceeding the limit receive `429 Too Many Requests` with a `Retry-After` header.
