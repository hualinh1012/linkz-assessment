## 2.1 Purpose

This section walks through the **end-to-end runtime flows** that define the Linkz Seat Reservation Platform. Where the [1. General](./1.%20General.md) section defines the static contracts (schema, API shapes, configuration), this section describes the *dynamic behaviour* — the ordered sequence of steps, actor interactions, state transitions, and error paths for each critical user journey.

Each flow document cross-references the relevant General specs rather than restating them, so the contracts have a single source of truth.

## 2.3 Actors

| Actor | Role in flows |
|-------|---------------|
| **Browser (Client)** | Renders SSR pages, initiates API calls, follows redirects to Keycloak / AXS |
| **Next.js App** | SSR pages + Route Handlers under `src/app/api/`; JWT middleware in `src/middleware.ts` |
| **Keycloak** | Identity provider — login UI, token issuance, JWKS |
| **AXS Checkout** | Hosted payment page + webhook + transaction/void/refund APIs |
| **PostgreSQL** | System of record for seats, reservations, payments — also serves seat-availability reads directly (no cache at this stage) |

## 2.4 Cross-Cutting Rules (apply to every flow)

These invariants from [1. General](./1.%20General.md) hold across all flows below:

- A seat becomes `RESERVED` **only after payment is confirmed** — never at selection time.
- Reservation writes run inside a `SELECT FOR UPDATE` transaction; partial unique indexes are the last-resort guard against double-booking and multiple active reservations per user — see [1.3. Database Schema §1.4–§1.5](./1.3.%20Database%20Schema.md).
- Seat availability is read **directly from PostgreSQL** — there is no cache at this stage, so every read is fresh and there is no invalidation to maintain (see [SAD/2.3. ADR - Caching](../SAD/2.3.%20ADR%20-%20Caching.md)).
- Payment outcomes are **always server-verified** — browser redirect parameters are never trusted alone (see [1.6. Payment Integration §4.6](./1.6.%20Payment%20Integration.md)).
