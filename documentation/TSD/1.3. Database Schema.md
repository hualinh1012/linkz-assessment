## Purpose

This document defines the complete relational schema for the Linkz Seat Reservation Platform. It covers table definitions, field types, constraints, indexes, enum values, and seed data. The schema is implemented via **Prisma ORM** and targets **PostgreSQL** (ap-southeast-1).

## 1.2 Prisma Schema

```prisma
// prisma/schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

// ─── Enums ────────────────────────────────────────────────────────────────────

enum SeatStatus {
  AVAILABLE
  PENDING
  RESERVED
}

enum ReservationStatus {
  PENDING
  CONFIRMED
  CANCELLED
}

enum PaymentStatus {
  INITIATED
  SUCCESS
  FAILED
  EXPIRED
  VOIDED
  REFUNDED
}

// ─── Tables ───────────────────────────────────────────────────────────────────

model User {
  id          String   @id @default(uuid())
  keycloakId  String   @unique @map("keycloak_id")
  email       String   @unique
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")

  reservations Reservation[]

  @@map("users")
}

model Seat {
  id        Int        @id
  label     String     @unique
  status    SeatStatus @default(AVAILABLE)
  updatedAt DateTime   @updatedAt @map("updated_at")

  reservations Reservation[]

  @@map("seats")
}

model Reservation {
  id        String            @id @default(uuid())
  userId    String            @map("user_id")
  seatId    Int               @map("seat_id")
  status    ReservationStatus @default(PENDING)
  expiresAt DateTime          @map("expires_at")
  createdAt DateTime          @default(now()) @map("created_at")
  updatedAt DateTime          @updatedAt @map("updated_at")

  user    User     @relation(fields: [userId], references: [id])
  seat    Seat     @relation(fields: [seatId], references: [id])
  payment Payment?

  @@index([userId])
  @@index([seatId])
  @@index([status])
  @@index([expiresAt])
  @@map("reservations")
}

model Payment {
  id                String        @id @default(uuid())
  reservationId     String        @unique @map("reservation_id")
  axsTransactionRef String?       @map("axs_transaction_ref")
  status            PaymentStatus @default(INITIATED)
  amountCents       Int           @map("amount_cents")
  currency          String        @default("SGD")
  initiatedAt       DateTime      @default(now()) @map("initiated_at")
  completedAt       DateTime?     @map("completed_at")

  reservation Reservation @relation(fields: [reservationId], references: [id])

  @@map("payments")
}
```

## 1.3 Table Definitions

### users

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `UUID` | PK, default `gen_random_uuid()` | Internal user identifier |
| `keycloak_id` | `TEXT` | NOT NULL, UNIQUE | Keycloak `sub` claim — source of identity |
| `email` | `TEXT` | NOT NULL, UNIQUE | Email mirrored from Keycloak JWT for display |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, default `now()` | Account first seen |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | Auto-updated on any row change |

### seats

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `INTEGER` | PK | Fixed IDs: 1, 2, 3 |
| `label` | `TEXT` | NOT NULL, UNIQUE | Human-readable label: "Seat A", "Seat B", "Seat C" |
| `status` | `seat_status` | NOT NULL, default `AVAILABLE` | Current availability state |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | Auto-updated on status change |

### reservations

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `UUID` | PK, default `gen_random_uuid()` | Reservation identifier |
| `user_id` | `UUID` | NOT NULL, FK → `users.id` | Who holds this reservation |
| `seat_id` | `INTEGER` | NOT NULL, FK → `seats.id` | Which seat is reserved |
| `status` | `reservation_status` | NOT NULL, default `PENDING` | Lifecycle state |
| `expires_at` | `TIMESTAMPTZ` | NOT NULL | When PENDING expires (15 min from creation) |
| `created_at` | `TIMESTAMPTZ` | NOT NULL, default `now()` | When the reservation was initiated |
| `updated_at` | `TIMESTAMPTZ` | NOT NULL | Auto-updated on any state change |

### payments

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `UUID` | PK, default `gen_random_uuid()` | Payment record identifier |
| `reservation_id` | `UUID` | NOT NULL, UNIQUE, FK → `reservations.id` | One payment per reservation |
| `axs_transaction_ref` | `TEXT` | NULLABLE | AXS transaction ID (set after AXS processes payment) |
| `status` | `payment_status` | NOT NULL, default `INITIATED` | AXS payment lifecycle state. `EXPIRED` = abandoned/timed-out before completion (set by the expiry cleanup, see [2.5. Expired Payment Clean Up](./2.5.%20Expired%20Payment%20Clean%20Up.md)); `FAILED` = AXS declined the payment |
| `amount_cents` | `INTEGER` | NOT NULL | Charge amount in cents (e.g. 1000 = SGD 10.00) |
| `currency` | `TEXT` | NOT NULL, default `SGD` | ISO 4217 currency code |
| `initiated_at` | `TIMESTAMPTZ` | NOT NULL, default `now()` | When user was redirected to AXS |
| `completed_at` | `TIMESTAMPTZ` | NULLABLE | When AXS webhook confirmed outcome |

## 1.4 Concurrency Constraints

Two partial unique indexes enforce the core business rules at the database level — these cannot be expressed in Prisma schema directly and are applied via a raw migration.

### One active reservation per user

A user may not hold more than one PENDING or CONFIRMED reservation at any time.

```sql
CREATE UNIQUE INDEX uq_user_active_reservation
  ON reservations (user_id)
  WHERE status IN ('PENDING', 'CONFIRMED');
```

### One active reservation per seat

A seat may not be claimed by more than one active reservation at any time.

```sql
CREATE UNIQUE INDEX uq_seat_active_reservation
  ON reservations (seat_id)
  WHERE status IN ('PENDING', 'CONFIRMED');
```

Both indexes are enforced inside a `SELECT FOR UPDATE` transaction (see [2.2. ADR - Database](../SAD/2.2.%20ADR%20-%20Database.md)) — the DB lock guarantees atomicity; the partial unique index is the last-resort guard.

## 1.5 Reservation Write Transaction

The reservation creation path runs inside a single transaction. Concurrency safety comes from the `SELECT ... FOR UPDATE` row lock (under the default `READ COMMITTED` isolation); it is not a `SERIALIZABLE` transaction. The new reservation's id is captured via `RETURNING` so the payment row can reference it:

```sql
BEGIN;

-- Lock the seat row — blocks concurrent reservation attempts on the same seat
SELECT id, status FROM seats WHERE id = $1 FOR UPDATE;

-- Reject if seat is not AVAILABLE
-- (application raises error if status != 'AVAILABLE')

-- Create reservation with 15-minute expiry; return the generated id
INSERT INTO reservations (id, user_id, seat_id, status, expires_at)
VALUES (gen_random_uuid(), $2, $1, 'PENDING', now() + interval '15 minutes')
RETURNING id;   -- captured by the app as :reservation_id

-- Mark seat as PENDING
UPDATE seats SET status = 'PENDING', updated_at = now() WHERE id = $1;

-- Create associated payment record, referencing the reservation just created
INSERT INTO payments (id, reservation_id, amount_cents, currency)
VALUES (gen_random_uuid(), :reservation_id, $3, 'SGD');

COMMIT;
```

> `:reservation_id` is the value returned by the reservation `INSERT`. Equivalently, the application can generate the reservation UUID itself and pass it as a bind parameter to both inserts — either way the payment must reference the actual reservation id, not a free `$3`.

## 1.6 PENDING Expiry Cleanup

PENDING reservations that exceed 15 minutes are released by a scheduled job. The job runs every minute via a Next.js cron route or an external AWS EventBridge rule.

The whole cleanup runs in a single transaction driven by **one snapshot of the expired set**, so payment, reservation, and seat transitions all act on exactly the same rows. Seat release is keyed on that snapshot — never on a time-window heuristic — so it can neither miss an expired seat nor collide with seats released by the payment-callback path:

```sql
BEGIN;

-- Snapshot the reservations that have expired in this run
WITH expired AS (
  SELECT id, seat_id
  FROM reservations
  WHERE status = 'PENDING'
    AND expires_at < now()
  FOR UPDATE
)
-- 1. Expire their orphaned (still-INITIATED) payments
, _payments AS (
  UPDATE payments
  SET status = 'EXPIRED', completed_at = now()
  WHERE status = 'INITIATED'
    AND reservation_id IN (SELECT id FROM expired)
  RETURNING reservation_id
)
-- 2. Cancel the expired reservations
, _reservations AS (
  UPDATE reservations
  SET status = 'CANCELLED', updated_at = now()
  WHERE id IN (SELECT id FROM expired)
  RETURNING seat_id
)
-- 3. Release exactly those seats
UPDATE seats
SET status = 'AVAILABLE', updated_at = now()
WHERE id IN (SELECT seat_id FROM expired);

COMMIT;
```

All three writes reference the `expired` CTE, so the seat release is tied to the reservations actually expired in this run rather than to a fragile `updated_at > now() - interval '1 minute'` window.

There is no cache at this stage, so released seats are reflected on the next read directly from PostgreSQL — no invalidation step is required (see [SAD/2.3. ADR - Caching](../SAD/2.3.%20ADR%20-%20Caching.md)). The end-to-end behaviour of this job — detection, payment transition, and seat release — is described in [2.5. Expired Payment Clean Up](./2.5.%20Expired%20Payment%20Clean%20Up.md).

## 1.7 Seed Data

The platform starts with exactly 3 seats. Seed data is applied via `prisma db seed`.

```typescript
// prisma/seed.ts
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();

async function main() {
  await prisma.seat.createMany({
    data: [
      { id: 1, label: 'Seat A', status: 'AVAILABLE' },
      { id: 2, label: 'Seat B', status: 'AVAILABLE' },
      { id: 3, label: 'Seat C', status: 'AVAILABLE' },
    ],
    skipDuplicates: true,
  });
}

main()
  .catch(console.error)
  .finally(() => prisma.$disconnect());
```

Seat IDs are fixed integers (1, 2, 3). The schema is seat-count agnostic — adding more seats requires only additional seed rows, no schema migration.

## 1.8 Migration Strategy

| Phase | Approach |
|-------|----------|
| Development | `prisma migrate dev` — generates and applies migrations locally |
| Staging | `prisma migrate deploy` — applies pending migrations on startup via ECS task entrypoint |
| Production | `prisma migrate deploy` — applied as a pre-deploy step in GitHub Actions before new ECS tasks start (zero downtime for additive migrations) |

Destructive migrations (column drops, type changes) must be run in two releases: first make the column nullable / add the new column; second release removes the old column after code no longer references it.
