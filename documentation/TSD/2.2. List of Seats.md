# 2.2. List of Seats

**Status:** DRAFT  
**Date:** 2026-06-17  
**Authors:** HuaHoangLinh

## Purpose

This document describes the flow for viewing seat availability — the only public, unauthenticated journey in the platform. It covers server-side rendering of the public page and the direct-from-PostgreSQL read path. The endpoint contract is defined in [1.4. API Specification §2.4 (GET /api/public/seats)](./1.4.%20API%20Specification.md#get-apipublicseats).

---

## 1. Flow Overview

Any visitor — logged in or not — can see the current status of all 3 seats. The public seat page is **server-side rendered** so unauthenticated visitors get real-time availability without client-side JavaScript fetches.

```
Browser (guest or authenticated)
   │
   │  GET /  (public seat page)
   ▼
Next.js Server Component (SSR)
   │
   │  internally reads seat availability
   ▼
GET /api/public/seats  ── PUBLIC, no JWT (bypassed by middleware)
   │
   ▼
Read directly from PostgreSQL (see §2)
   │
   ▼
Render seats with status badges:
   AVAILABLE → selectable (CTA: "Reserve")
   PENDING   → shown as temporarily held
   RESERVED  → shown as taken
   │
   └─ all seats RESERVED/PENDING → "Sold Out" state
```

The page is publicly reachable; `GET /api/public/seats` lives under the `/api/public/` prefix — the middleware bypasses the entire prefix, so no token is required (see [1.4 §2.6](./1.4.%20API%20Specification.md)).

---

## 2. Read Path — Direct from PostgreSQL

There is **no cache at this stage** (see [SAD/2.3. ADR - Caching](../SAD/2.3.%20ADR%20-%20Caching.md)). With only 3 seats, availability is a sub-millisecond indexed read straight from PostgreSQL — a cache would optimise a non-bottleneck and add invalidation risk.

```
GET /api/public/seats
   │
   ▼
Prisma → SELECT id, label, status FROM seats ORDER BY id
   │
   ▼
return seats[]
```

- The read goes straight to PostgreSQL on every request; the result is **always fresh** because the DB is the single source of truth.
- No invalidation logic exists, so the reservation/payment/expiry flows carry no "evict cache" step.
- When read volume eventually justifies it, a Redis cache is introduced behind a `CacheProvider` port (cache-aside, write-through eviction on mutations) — purely additive, per the ADR.

---

## 3. Response

Per [1.4 §2.4](./1.4.%20API%20Specification.md):

```jsonc
// 200 OK
{
  "seats": [
    { "id": 1, "label": "Seat A", "status": "AVAILABLE" },
    { "id": 2, "label": "Seat B", "status": "PENDING" },
    { "id": 3, "label": "Seat C", "status": "RESERVED" }
  ]
}
```

The `status` enum (`AVAILABLE | PENDING | RESERVED`) is defined in [1.3. Database Schema](./1.3.%20Database%20Schema.md).

---

## 4. Rate Limiting

`GET /api/public/seats` is limited to **60 requests / minute per IP** (see [1.4 §2.7](./1.4.%20API%20Specification.md)). Requests over the limit receive `429 Too Many Requests` with a `Retry-After` header. Rate-limit state is held in-process per instance at this stage (no shared cache yet — moves to Redis at scale).

---

## 5. Edge Cases

| Scenario | Outcome |
|----------|---------|
| Stale availability | Cannot occur — every read hits PostgreSQL directly; there is no cache that could drift |
| All seats unavailable | Page renders the **"Sold Out"** state; no reserve CTA |
| PENDING seat whose reservation has just expired | The per-minute cleanup job releases it to `AVAILABLE` (see [1.3 §1.6](./1.3.%20Database%20Schema.md)); the next read reflects availability immediately |
| Unauthenticated visitor clicks "Reserve" | Redirected into the [2.1. Authentication](./2.1.%20Authentication.md) login flow before any reservation is attempted |
