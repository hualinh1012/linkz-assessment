## Purpose

| Item | Value |

| Database | PostgreSQL — same RDS instance, separate `keycloak_db` database |
| ECS tasks | 2 (Production), 1 (Staging) |

## 3.2 Realm Configuration

A dedicated realm named `linkz` is created in Keycloak. All configuration below applies to this realm.

### Session Lifetime

| Setting | Value | Path in Keycloak Admin |
|---------|-------|----------------------|
| SSO Session Max | `7776000` seconds (90 days) | Realm Settings → Sessions → SSO Session Max |
| SSO Session Idle | `7776000` seconds (90 days) | Realm Settings → Sessions → SSO Session Idle |
| Access Token Lifespan | `900` seconds (15 minutes) | Realm Settings → Tokens → Access Token Lifespan |
| Refresh Token Max Reuse | `0` (single-use rotation) | Realm Settings → Tokens → Refresh Token Max Reuse |

These Keycloak settings govern the OIDC login session. The app itself, however, does **not** use the Keycloak access/refresh tokens per request — Auth.js issues its own 90-day encrypted session cookie at login (see [2.1 §2](./2.1.%20Authentication.md)). The 15-minute access-token lifespan therefore only bounds the short-lived token exchanged during the Auth.js handshake; the user stays logged in for the 90-day cookie window without re-authenticating.

### Password Policy

| Policy | Setting |
|--------|---------|
| Minimum length | 8 characters |
| Special characters | 1 required |
| Digits | 1 required |
| Not username | Enabled |
| Password history | 3 previous passwords |

## 3.3 Client Configuration

A single client named `linkz-web` is registered in the `linkz` realm.

| Setting | Value |
|---------|-------|
| Client ID | `linkz-web` |
| Client Protocol | `openid-connect` |
| Access Type | `confidential` |
| Standard Flow | Enabled (Authorization Code flow) |
| Direct Access Grants | Disabled |
| Root URL | `https://linkz.example.com` |
| Valid Redirect URIs | `https://linkz.example.com/*` |
| Valid Post Logout Redirect URIs | `https://linkz.example.com/` |
| Web Origins | `https://linkz.example.com` |
| PKCE Code Challenge Method | `S256` |

### Client Scopes

| Scope | Included in Token |
|-------|------------------|
| `openid` | Yes |
| `profile` | Yes |
| `email` | Yes |
| `offline_access` | Yes — enables refresh token |

### Audience Mapper (required)

By default Keycloak does **not** place the client ID in the access token's `aud` claim (it typically emits `aud: "account"`). Since the application verifies `audience: linkz-web` (§3.5), an **Audience mapper must be configured** or every otherwise-valid token is rejected.

Add a dedicated client scope `linkz-audience` (assigned as a default scope to `linkz-web`) with a protocol mapper of type **Audience**, *Included Client Audience* = `linkz-web`, *Add to access token* = ON. This makes the access token carry `aud: linkz-web`, satisfying the JWKS verification below.

## 3.4 JWT Claims

The Keycloak-issued JWT contains the following claims used by the application:

| Claim | Type | Description |
|-------|------|-------------|
| `sub` | `string` | Keycloak user UUID — used as `keycloak_id` in the `users` table |
| `email` | `string` | User email address |
| `email_verified` | `boolean` | Whether email has been verified |
| `preferred_username` | `string` | Username (often same as email) |
| `iss` | `string` | Token issuer — must match `KEYCLOAK_ISSUER` env var |
| `aud` | `string` | Audience — must include `linkz-web` |
| `exp` | `number` | Expiry timestamp |
| `iat` | `number` | Issued-at timestamp |

The application verifies `iss`, `aud`, and `exp` on every request via JWKS. No additional claims are required at early stage.

## 3.5 Session Verification in Next.js (Auth.js)

The OIDC handshake and session are handled by **Auth.js (NextAuth v5)** with the Keycloak provider — the app does not hand-roll the PKCE flow or JWKS verification. Configuration is split so the Edge middleware never imports Prisma:

```typescript
// src/auth.config.ts  (edge-safe — shared by middleware and the Node instance)
import Keycloak from "next-auth/providers/keycloak";

export const authConfig = {
  providers: [Keycloak({ clientId, clientSecret, issuer })], // from env
  session: { strategy: "jwt", maxAge: 60 * 60 * 24 * 90 },   // 90 days
  callbacks: {
    jwt({ token, profile }) { if (profile?.sub) token.sub = profile.sub; return token; },
    session({ session, token }) { if (token.sub) session.user.id = token.sub; return session; },
  },
} satisfies NextAuthConfig;

// src/auth.ts  (node — adds the Prisma-backed sign-in provisioning, §3.6)
export const { handlers, auth, signIn, signOut } = NextAuth({ ...authConfig, events: { signIn } });
```

The session is stored in an encrypted HTTP-only cookie (no session table). `src/middleware.ts` builds an Edge Auth.js instance from `authConfig`, reads the session, and forwards `x-user-id` (the `sub`) to Route Handlers (see [1.4 §2.6](./1.4.%20API%20Specification.md)). Auth.js validates the Keycloak ID token against JWKS internally during the handshake — no manual key management.

> Requires `AUTH_SECRET` (cookie encryption) and `KEYCLOAK_CLIENT_SECRET`. The Keycloak client's Valid Redirect URIs must include `<app>/api/auth/callback/keycloak`. The Audience mapper in §3.3 is still recommended so the Keycloak access/ID token carries `aud=linkz-web`.

## 3.6 User Provisioning on First Login

On first sign-in, the Auth.js `signIn` event (Node runtime) provisions a local `users` record linked by `keycloak_id`, via the `ProvisionUser` use case → `UserRepository.upsertByKeycloakId`:

```typescript
// src/auth.ts — events.signIn
async signIn({ profile }) {
  if (profile?.sub && typeof profile.email === "string") {
    await container.provisionUser.execute({ keycloakId: profile.sub, email: profile.email });
  }
}

// UserRepository adapter (Prisma)
upsert({ where: { keycloakId }, update: { email }, create: { keycloakId, email } });
```

The upsert is idempotent, so concurrent first sign-ins are safe.

## 3.7 Keycloakify Login Theme

Keycloakify replaces Keycloak's default Freemarker login pages with a custom React + TypeScript + Tailwind theme. The theme lives in the same Next.js monorepo under `src/keycloak-theme/`.

### Theme Build & Deployment

| Step | Command / Action |
|------|-----------------|
| Build theme `.jar` | `npx keycloakify build` in project root |
| Output | `dist_keycloak/keycloak-theme-<version>.jar` |
| Deployment | `.jar` is stored on EFS; mounted into the Keycloak ECS task at `/opt/keycloak/providers/` |
| Theme activation | Realm Settings → Themes → Login Theme → `linkz` |

### Theme Directory Structure

```
src/keycloak-theme/
├── login/
│   ├── pages/
│   │   ├── Login.tsx          # Email + password form
│   │   ├── Register.tsx       # Registration form
│   │   └── Error.tsx          # Error page
│   ├── KcPage.tsx             # Entry point — receives kcContext
│   └── index.ts
└── shared/
    └── components/            # Shared UI — reuses design system from main app
```

### Shared Design System

The Keycloakify theme imports the same Tailwind configuration and shared UI components as the main application. This ensures:
- Consistent brand identity between the login page and the main app.
- A single Tailwind config file governs all styling.
- No separate CSS framework or design token duplication.

## 3.9 Session Revocation — Keycloak Back-Channel Logout

Because the Auth.js session cookie is stateless (validated locally with `AUTH_SECRET`), it cannot be force-invalidated per-user without a server-side denylist. Keycloak Back-Channel Logout provides the revocation signal.

### Configuration in Keycloak

On the `linkz-web` client, set:

| Setting | Value |
|---------|-------|
| Back-Channel Logout URL | `https://linkz.example.com/api/auth/backchannel-logout` |
| Back-Channel Logout Session Required | `ON` |
| Back-Channel Logout Revoke Offline Sessions | `ON` |

Keycloak will POST a signed **Logout Token** (JWS) to this URL whenever a session is revoked — triggered by password change, admin session revocation, or user disable.

### Back-Channel Logout Endpoint

`POST /api/auth/backchannel-logout` (lives under `/api/auth/`, bypassed by the JWT middleware):

1. Verify the Logout Token (JWS) against Keycloak's JWKS endpoint.
2. Extract the `sub` claim from the token.
3. Upsert `sub` into the `revoked_sessions` table (see [1.3 §1.7](./1.3.%20Database%20Schema.md)).
4. Return `200 OK`.

### Middleware Denylist Check

After decrypting the Auth.js session cookie, `src/middleware.ts` performs a single indexed PK lookup on `revoked_sessions` for the session's `sub`. A match returns `401 UNAUTHORIZED` before any handler logic runs (see [1.4 §2.6](./1.4.%20API%20Specification.md)).

### Scale Path

At scale, the `revoked_sessions` PostgreSQL table is replaced by a Redis denylist (ElastiCache — already in the target architecture). The middleware check is a key lookup regardless of backend; no logic change is required when switching.

## 3.8 Social Login — Long-Term Configuration

The following identity providers will be added to the `linkz` realm when the long-term roadmap is activated. No application code changes are required — Keycloak handles the IdP federation.

| Provider | Keycloak Identity Provider | Required Credentials |
|----------|---------------------------|---------------------|
| Google SSO | `google` | Google OAuth2 Client ID + Secret |
| Facebook SSO | `facebook` | Facebook App ID + Secret |
| Apple SSO | `apple` (via OIDC) | Apple Services ID + Key |
| Phone / OTP | Keycloak SMS SPI | SMS gateway credentials (e.g. AWS SNS) |
| Guest checkout | Custom Keycloak SPI | No external credentials |

Each integration is an additive configuration change to the Keycloak realm — not a code change in the Next.js application.
