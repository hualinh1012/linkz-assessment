# 2.3. Reserve Seat

**Status:** DRAFT  
**Date:** 2026-06-17  
**Authors:** HuaHoangLinh

## Purpose

This document describes the seat reservation flow — what happens when an authenticated user selects an available seat. This is the most concurrency-sensitive path in the platform: it must prevent double-booking and enforce one active reservation per user, then hand off to payment. The endpoint contract is in [1.4. API Specification (POST /api/reservations)](./1.4.%20API%20Specification.md); the transaction and constraints are in [1.3. Database Schema §1.4–§1.5](./1.3.%20Database%20Schema.md).

---

## 1. Flow Overview

```
Authenticated user clicks "Reserve" on an AVAILABLE seat
   │
   │  POST /api/reservations  { seatId }   (Authorization: Bearer …)
   ▼
src/middleware.ts → JWT verified (see 2.1. Authentication)
   ▼
Rate limit check (5 req/min/user)
   ▼
┌──────────── DB transaction (SELECT FOR UPDATE) ────────────┐
│ 1. Lock seat row: SELECT … FROM seats WHERE id=$1 FOR UPDATE│
│ 2. Reject if seat.status != AVAILABLE          → 409        │
│ 3. Reject if user has an active reservation     → 409        │
│ 4. INSERT reservation (PENDING, expires_at = now()+15min)   │
│ 5. UPDATE seat → PENDING                                    │
│ 6. INSERT payment (INITIATED, amount, SGD)                  │
└──────────────────────────────── COMMIT ────────────────────┘
   ▼
Build JWE payment link URL (see 1.6 §4.2)
   ▼
201 Created → { reservation, payment, redirectUrl }
   ▼
Client redirects browser to redirectUrl  → continues in 2.4. Payment & Callback
```

---

## 2. Concurrency Control

This flow is where double-booking is prevented. Two layers guard it (see [1.3 §1.4–§1.5](./1.3.%20Database%20Schema.md)):

1. **Pessimistic lock** — `SELECT … FOR UPDATE` on the seat row. Concurrent reservation attempts on the same seat block until the first transaction commits or rolls back. The loser re-reads `status = PENDING` and is rejected.
2. **Partial unique indexes** (last-resort guard, enforced at DB level):
   - `uq_seat_active_reservation` — at most one `PENDING`/`CONFIRMED` reservation per seat.
   - `uq_user_active_reservation` — at most one `PENDING`/`CONFIRMED` reservation per user.

The full write transaction is specified in [1.3 §1.5](./1.3.%20Database%20Schema.md):

```sql
BEGIN;
SELECT id, status FROM seats WHERE id = $1 FOR UPDATE;   -- lock
-- app rejects if status != 'AVAILABLE'
INSERT INTO reservations (…, status, expires_at)
  VALUES (…, 'PENDING', now() + interval '15 minutes');
UPDATE seats SET status = 'PENDING', updated_at = now() WHERE id = $1;
INSERT INTO payments (…, status) VALUES (…, 'INITIATED');
COMMIT;
```

## 3. State After a Successful Reservation

| Entity | Before | After |
|--------|--------|-------|
| `seats.status` | `AVAILABLE` | `PENDING` |
| `reservations.status` | — | `PENDING` (`expires_at = now() + 15 min`) |
| `payments.status` | — | `INITIATED` |

(No cache at this stage — the next `GET /api/seats` reflects `PENDING` directly from PostgreSQL.)

The seat is **held, not reserved** — it only becomes `RESERVED` after payment confirmation in [2.4. Payment & Callback](./2.4.%20Payment%20%26%20Callback.md). If payment is not completed within 15 minutes, the seat is released (see §5).

## 4. Response

Per [1.4 §2.4](./1.4.%20API%20Specification.md):

```jsonc
// 201 Created
{
  "reservation": { "id": "<uuid>", "seatId": 1, "status": "PENDING", "expiresAt": "<ISO-8601>" },
  "payment":     { "id": "<uuid>", "status": "INITIATED" },
  "redirectUrl": "<signed AXS Payment Page URL>"
}
```

The client redirects the browser to `redirectUrl`. The AXS URL is a JWE-encrypted payment link embedding `paymentId` as `merchantRef` plus `successUrl`, `failUrl`, and `webhookUrl` (see [1.6 §4.2](./1.6.%20Payment%20Integration.md)).

## 5. PENDING Expiry

A `PENDING` reservation that is not paid within **15 minutes** is released by the per-minute cleanup job (see [1.3 §1.6](./1.3.%20Database%20Schema.md)):

```
expires_at < now()  →  reservation: PENDING → CANCELLED
                        seat:        PENDING → AVAILABLE
```

The 15-minute window is aligned with the AXS Checkout maximum payment-page session, so a user can never be on a valid payment page after their hold has expired (see [1.6 §4.8](./1.6.%20Payment%20Integration.md)).

## 6. Error Cases

Per [1.4 §2.4](./1.4.%20API%20Specification.md):

| Scenario | HTTP | `code` |
|----------|------|--------|
| Seat does not exist | 404 | `NOT_FOUND` |
| Seat is not `AVAILABLE` (already PENDING/RESERVED) | 409 | `CONFLICT` |
| User already holds an active reservation | 409 | `CONFLICT` |
| AXS payment initiation failed | 502 | `PAYMENT_INITIATION_FAILED` |
| Rate limit exceeded (>5/min/user) | 429 | — (`Retry-After` header) |
| Missing/invalid JWT | 401 | `UNAUTHORIZED` |

If AXS payment initiation fails **after** the DB transaction has committed, the reservation exists as `PENDING` and will be released by the 15-minute expiry job — the user can retry from the seat page.
