# 2.6. Test — Free Up All Seats

**Status:** DRAFT  
**Date:** 2026-06-17  
**Authors:** HuaHoangLinh

---

## Purpose

This document describes a **test-only backdoor** that resets every seat back to `AVAILABLE` in a single call, so manual and automated tests can return the system to a clean baseline without manually unwinding reservations and payments.

> ⚠️ **This endpoint is intentionally NOT part of the public [1.4. API Specification](./1.4.%20API%20Specification.md).** It is a development/testing affordance, not a product feature. It must never be reachable in production. It is documented here only so the test harness and reviewers know it exists and how it is guarded.

---

## 1. Endpoint

```
POST /api/public/test/reset-seats
```

Resets all seats and any associated reservations/payments to a clean state in one transaction.

There is no request body. Authentication is **not** via the normal Keycloak JWT (this is below the product auth layer) — it is gated by the guards in §3 instead.

---

## 2. Behaviour

```
POST /api/public/test/reset-seats
   │
   ▼
Guard check (see §3) — else 404 Not Found
   │
   ▼
BEGIN
   │  1. payments:     INITIATED/SUCCESS for active reservations → EXPIRED (completed_at = now())
   │  2. reservations: PENDING/CONFIRMED → CANCELLED
   │  3. seats:        any status → AVAILABLE
COMMIT
   │
   ▼
200 OK { reset: true, seatsFreed: <n> }
```

This is a deliberately blunt reset: it does **not** call AXS void/refund (it is a local DB reset for test fixtures only). It mirrors the terminal states used by [2.5. Expired Payment Clean Up](./2.5.%20Expired%20Payment%20Clean%20Up.md) so the resulting data is still schema-valid per [1.3. Database Schema](./1.3.%20Database%20Schema.md).

## 3. Guards (mandatory)

The endpoint must be **inaccessible in production**. Both guards apply:

1. **Environment gate** — the handler is a no-op unless `NODE_ENV !== 'production'`. In production it returns `404 Not Found` (indistinguishable from an unmapped route), so the backdoor is invisible.
2. **Shared-secret header** — requests must carry a secret test token, e.g. `X-Test-Token: <TEST_RESET_TOKEN>`, matched against an env var that is **only set in development/staging** (see [1.7. Environment & Configuration](./1.7.%20Environment%20%26%20Configuration.md)). Missing/incorrect token → `404 Not Found`.

```typescript
// app/api/public/test/reset-seats/route.ts  (test backdoor — guarded)
export async function POST(req: Request) {
  if (process.env.NODE_ENV === 'production') return new Response(null, { status: 404 });
  if (req.headers.get('x-test-token') !== process.env.TEST_RESET_TOKEN) {
    return new Response(null, { status: 404 });
  }
  const seatsFreed = await container.resetAllSeats.execute();   // test-only use case
  return Response.json({ reset: true, seatsFreed });
}
```

The route is **not** added to the middleware `PUBLIC_PATHS` allow-list as a product route; its guards are self-contained. Ideally the route file is excluded from the production build entirely.

---

## 4. Usage

```bash
# Local / staging only
curl -X POST http://localhost:3000/api/public/test/reset-seats \
  -H "X-Test-Token: $TEST_RESET_TOKEN"
# → { "reset": true, "seatsFreed": 3 }
```

Typical use is a test-suite `beforeEach`/`afterEach` hook, or a manual reset while exercising the reservation and payment flows.

---

## 5. Why This Is Safe to Keep in the Repo

| Concern | Mitigation |
|---------|------------|
| Exposed in production | Hard `NODE_ENV !== 'production'` gate returns 404; route preferably stripped from the prod build |
| Discoverable / brute-forced | Requires a secret `X-Test-Token`; returns 404 (not 401/403) so its existence is not revealed |
| Confused with product API | Deliberately omitted from [1.4. API Specification](./1.4.%20API%20Specification.md); lives under `/api/public/test/` with its own double guard |
| Leaves invalid data | Runs in one transaction, uses the same terminal states as the real flows; next read reflects the reset directly (no cache) |
