# 2.4. Payment & Callback

**Status:** DRAFT

## Purpose

This document describes what happens after a user is redirected to the AXS Payment Page: how payment outcomes are received and verified, how the seat transitions to `RESERVED` (or is released), and how cancellation/refund works. It is the runtime view of [1.6. Payment Integration](./1.6.%20Payment%20Integration.md) and the `POST /api/public/payment/callback` + `DELETE /api/reservations/:id` contracts in [1.4. API Specification](./1.4.%20API%20Specification.md).

This flow continues directly from [2.3. Reserve Seat](./2.3.%20Reserve%20Seat.md), which left the seat `PENDING`, the payment `INITIATED`, and returned a `redirectUrl`.

All public routes — including the webhook — live under `/api/public/` (see [1.4 §2.2](./1.4.%20API%20Specification.md)).

## 1. Sequence Diagram

```mermaid
sequenceDiagram
    actor User
    participant App as Application (Next.js)
    participant DB as PostgreSQL
    participant AXS as AXS Checkout

    User->>App: POST /api/reservations { seatId }
    App->>App: Verify JWT (middleware)
    App->>DB: BEGIN; SELECT seat FOR UPDATE
    DB-->>App: seat row (AVAILABLE)
    App->>DB: INSERT reservation (PENDING, expires_at=now()+15min)
    App->>DB: UPDATE seat → PENDING
    App->>DB: INSERT payment (INITIATED)
    App->>DB: COMMIT
    App->>App: Encrypt JWE payment link (§4.2)
    App-->>User: 201 { redirectUrl, reservation, payment }

    User->>AXS: GET redirectUrl → AXS Payment Page
    Note over User,AXS: User selects payment method and pays

    alt Payment SUCCESS
        AXS-->>User: Redirect to successUrl (/reservation/confirmed?id=…)
        AXS->>App: POST /api/public/payment/callback  (JWE body)
        App->>App: Decrypt JWE, validate fields
        App->>DB: Idempotency check (payment already terminal?)
        App->>DB: BEGIN
        App->>DB: UPDATE payment  → SUCCESS (set transactionRef, completedAt)
        App->>DB: UPDATE reservation → CONFIRMED
        App->>DB: UPDATE seat → RESERVED
        App->>DB: COMMIT
        App-->>AXS: 200 { received: true }
        User->>App: GET /reservation/confirmed?id=…
        App->>DB: SELECT reservation (now CONFIRMED)
        App-->>User: Confirmation page
    else Payment DECLINED or EXPIRED
        AXS-->>User: Redirect to failUrl (/seats?error=payment_failed)
        AXS->>App: POST /api/public/payment/callback  (JWE body, status=DECLINED|EXPIRED)
        App->>App: Decrypt JWE, validate fields
        App->>DB: Idempotency check
        App->>DB: BEGIN
        App->>DB: UPDATE payment  → FAILED | EXPIRED (set completedAt)
        App->>DB: UPDATE reservation → CANCELLED
        App->>DB: UPDATE seat → AVAILABLE
        App->>DB: COMMIT
        App-->>AXS: 200 { received: true }
        User->>App: GET /seats?error=payment_failed
        App-->>User: Seat listing with error message
    end
```

## 2. Flow Overview

The payment result reaches the app via **two independent channels**:

| Channel | Role |
|---------|------|
| **(A) Browser redirect** to `successUrl` / `failUrl` | User navigation only — no DB state change |
| **(B) Webhook** `POST /api/public/payment/callback` | Authoritative server-to-server confirmation — drives all DB state transitions |

**AXS redirects the user first, then delivers the webhook.** The `successUrl`/`failUrl` pages show the appropriate UX immediately; the webhook (which typically arrives within seconds) makes the state official in the database. Never update DB state from the browser redirect URL.

---

## 3. Webhook Processing (authoritative)

`POST /api/public/payment/callback` — see [1.4 §2.4](./1.4.%20API%20Specification.md) and [1.6 §4.3–§4.5](./1.6.%20Payment%20Integration.md).

```
POST /api/public/payment/callback   (JWE-encrypted body)
   │
   ▼
1. Decrypt JWE body with AXS_API_SECRET (compactDecrypt)
      decryption fails → 401, before any DB access
   ▼
2. Validate required fields: transactionRef, merchantRef, status
      invalid → 400
   ▼
3. Idempotency: load payment by merchantRef (= paymentId)
      already terminal (SUCCESS/FAILED/EXPIRED/VOIDED/REFUNDED) → 200 { received: true }
   ▼
4. Apply state transition inside a DB transaction (§4)
   ▼
5. 200 OK { received: true }
```

## 4. State Transitions

### On `status = SUCCESS`

```
payments.status:              INITIATED → SUCCESS
payments.axs_transaction_ref: set to transactionRef
payments.completed_at:        set
reservations.status:          PENDING   → CONFIRMED
seats.status:                 PENDING   → RESERVED
```

`RESERVED` is the **only** path by which a seat reaches this state — never at selection time.

### On `status = DECLINED`

```
payments.status:       INITIATED → FAILED
payments.completed_at: set
reservations.status:   PENDING   → CANCELLED
seats.status:          PENDING   → AVAILABLE
```

### On `status = EXPIRED`

```
payments.status:       INITIATED → EXPIRED
payments.completed_at: set
reservations.status:   PENDING   → CANCELLED
seats.status:          PENDING   → AVAILABLE
```

`EXPIRED` occurs when the user does not complete payment within the AXS session timeout (15 min). This is the same terminal state the expiry cleanup job uses for reservations whose `expires_at` has passed (see [2.5](./2.5.%20Expired%20Payment%20Clean%20Up.md)).

## 5. Success and Fail Pages (browser)

### `GET /reservation/confirmed?id=<reservationId>`

The user lands here after a successful payment. Because the webhook may not have been processed yet when the browser arrives, the page:

1. Loads the reservation from DB by `id`.
2. If `status = CONFIRMED` → show full confirmation.
3. If `status = PENDING` → show "Payment received — confirming your booking..." (webhook is in flight; page can poll `GET /api/reservations/:id` until CONFIRMED).

### `GET /seats?error=payment_failed`

Standard seat listing page with an error banner. No special server logic needed — the `error` query param drives the client-side error message.

---

## 6. PENDING Expiry During Payment

The 15-minute `PENDING` window aligns with the AXS session timeout (see [1.6 §4.7](./1.6.%20Payment%20Integration.md)):

- If the user abandons the AXS page, AXS sends an `EXPIRED` webhook and redirects to `failUrl`.
- The expiry cleanup job (see [2.5](./2.5.%20Expired%20Payment%20Clean%20Up.md)) also picks up any reservations where `expires_at < now()` that did not receive a webhook (rare: network failure, dead-letter webhook).

## 7. Cancellation & Refund

A user may cancel a **CONFIRMED** reservation via `DELETE /api/reservations/:id` (see [1.4 §2.4](./1.4.%20API%20Specification.md)). This triggers an AXS void or refund and releases the seat.

```
DELETE /api/reservations/:id   (Authorization: Bearer …)
   │
   ▼
Verify JWT → reservation exists, owned by user, status = CONFIRMED
   ▼
reversePayment(axsTransactionRef, paymentId, amountCents)
  → attempt AXS void first
  → if void rejected (already settled) → attempt AXS refund
  (see 1.6 §4.6)
   ▼
On reversal success (DB transaction):
   reservations.status: CONFIRMED → CANCELLED
   payments.status:     SUCCESS   → VOIDED | REFUNDED
   seats.status:        RESERVED  → AVAILABLE
```

| Scenario | HTTP | `code` |
|----------|------|--------|
| Reservation not found | 404 | `NOT_FOUND` |
| Reservation belongs to another user | 403 | `FORBIDDEN` |
| Reservation not `CONFIRMED` | 409 | `CONFLICT` |
| AXS void + refund both failed | 502 | `PAYMENT_REVERSAL_FAILED` |

## 8. Security & Integrity Summary

| Control | Mechanism |
|---------|-----------|
| Webhook authenticity | JWE decryption with `AXS_API_SECRET`; reject on decryption failure before DB access |
| No trust in browser params | `successUrl`/`failUrl` are navigation-only; DB state is driven exclusively by the webhook |
| Duplicate webhooks | Idempotent — terminal payment states are not re-applied |
| No card data in scope | AXS hosts the payment page; the app never receives PAN/card data |
| State consistency | All transitions run in a single DB transaction; with no cache, the next read reflects committed state directly |
