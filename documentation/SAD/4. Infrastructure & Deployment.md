# 4. Infrastructure & Deployment

## 4.1 Purpose

This section describes the infrastructure topology, AWS service selection, environment strategy, and deployment approach for the Linkz Seat Reservation Platform. All components are deployed to **Amazon Web Services (AWS)** in the Singapore region (`ap-southeast-1`) to minimise latency for our target market and satisfy data residency expectations.

Detailed infrastructure-as-code specifications, instance sizing, and environment-specific configurations are covered in the Technical Specification Document (TSD).

---

## 4.2 AWS Region

| Decision | Value | Reason |
|----------|-------|--------|
| Primary region | `ap-southeast-1` (Singapore) | Closest to end users; aligns with AXS Checkout and MAS data residency expectations |
| Multi-AZ | Yes — across 2 Availability Zones minimum | Satisfies high-availability NFR; protects against single AZ failure |

## 4.4 AWS Services

| Service | Role |
|---------|------|
| **ECS Fargate** | Serverless container compute for Next.js app and Keycloak — no EC2 instances to manage |
| **ECR** | Private container registry for Docker images (Next.js, Keycloak + Keycloakify theme) |
| **ALB** | HTTPS load balancer routing traffic to Next.js and Keycloak ECS services |
| **RDS PostgreSQL** | Managed PostgreSQL with automated backups, point-in-time recovery, and Multi-AZ failover |
| **CloudFront** | CDN for PWA static assets and Next.js public files — global edge caching |
| **Route 53** | DNS management and health-check-based failover |
| **ACM** | SSL/TLS certificate provisioning and auto-renewal for all public domains |
| **Secrets Manager** | Secure storage for DB credentials, Keycloak secrets, AXS merchant credentials, Firebase service account key |
| **EFS** | Shared file storage for Keycloak theme (Keycloakify `.jar`) mounted across Keycloak ECS tasks |
| **VPC** | Network isolation with public and private subnets across two AZs |
| **CloudWatch** | Centralised logging, metrics, and alerting for all ECS services and RDS |

## 4.5 Network Architecture

```
VPC (ap-southeast-1)
├── Public Subnets (AZ-1, AZ-2)
│   ├── Application Load Balancer
│   └── NAT Gateway (outbound internet for private subnets)
│
└── Private Subnets (AZ-1, AZ-2)
    ├── ECS Fargate — Next.js App
    ├── ECS Fargate — Keycloak
    └── RDS PostgreSQL (Multi-AZ)
```

- All application workloads run in **private subnets** — not directly accessible from the internet.
- Only the ALB and CloudFront sit in public-facing positions.
- Outbound traffic from private subnets (e.g., to Firebase, AXS Checkout, Keycloak JWKS) routes through a **NAT Gateway**.
- Security groups restrict inter-service communication to only what is necessary (e.g., Next.js → RDS on PostgreSQL port only).

## 4.6 Environment Strategy

| Environment | Purpose | Notes |
|------------|---------|-------|
| **Development** | Local developer machines | Docker Compose runs Next.js, Keycloak, PostgreSQL locally; AXS sandbox credentials |
| **Staging** | Pre-production validation | Full AWS topology, reduced instance sizes; mirrors production config; AXS sandbox |
| **Production** | Live platform | Full AWS topology, Multi-AZ, auto-scaling enabled; AXS live credentials |

Each environment has its own:
- AWS account or isolated VPC
- Firebase project
- Keycloak realm
- RDS instance
- Secrets Manager entries

## 4.7 CI/CD Pipeline

```
Developer pushes to GitHub
          │
          ▼
    GitHub Actions
          │
    ┌─────┴──────┐
    │            │
  Lint +      Build
  Type check   Docker image
    │            │
    └─────┬──────┘
          │
          ▼
    Push to ECR
    (tagged with commit SHA)
          │
    ┌─────┴──────┐
    │            │
 Staging      Production
 (auto)       (manual approval)
    │            │
    ▼            ▼
ECS rolling  ECS rolling
update       update
```

- **Staging** deploys automatically on merge to `main`.
- **Production** requires a manual approval gate before the ECS rolling update proceeds.
- ECS rolling updates ensure zero downtime — old tasks remain live until new tasks pass health checks.
- Docker images are tagged with the Git commit SHA for full traceability.

## 4.8 High Availability

| Component | HA Strategy |
|-----------|------------|
| **Next.js App** | Multiple ECS Fargate tasks across AZ-1 and AZ-2; ALB distributes traffic; Hazelcast AWS discovery keeps cache consistent across tasks |
| **Keycloak** | Multiple ECS Fargate tasks across AZ-1 and AZ-2 behind ALB |
| **PostgreSQL** | RDS Multi-AZ — automatic standby failover in under 60 seconds |
| **Cache** | Hazelcast partitioned and replicated across Next.js tasks; backup partitions promoted automatically on task failure |
| **DNS** | Route 53 health checks with automatic failover |

## 4.9 Observability

| Concern | Approach |
|---------|---------|
| **Logging** | All ECS task stdout/stderr shipped to CloudWatch Logs; structured JSON logging from Next.js and Keycloak |
| **Metrics** | CloudWatch metrics for ECS CPU/memory, RDS connections and latency, ALB request count and error rates |
| **Alerting** | CloudWatch Alarms on error rate thresholds, RDS failover events, and ECS task failures |
| **Tracing** | AWS X-Ray for distributed tracing across Next.js API routes (future consideration) |
