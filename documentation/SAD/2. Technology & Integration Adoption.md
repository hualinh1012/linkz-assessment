# 2. Technology & Integration Adoption

## 2.1 Purpose

This document provides a consolidated view of all technology decisions made for the **Linkz Seat Reservation Platform**. Each decision is driven by the constraints and goals established in [1. High-level solution design](./1.%20High-level%20solution%20design.md) — startup talent pool alignment, time to market, operational simplicity, and scalability headroom.

Each technology area has a dedicated Architecture Decision Record (ADR) that documents the decision, the alternatives considered, and the rationale in full detail.

## 2.3 Decision Summary

### 2.3.1 Programming Language — TypeScript

TypeScript is our **primary and only language** across all layers — frontend, backend API, database schema types, and Keycloak theme (via Keycloakify). As a startup, our talent pool is TypeScript-first. A polyglot architecture would split our hiring funnel, increase onboarding time, and duplicate shared domain types across language boundaries.

`strict: true` is enforced to maximise compile-time safety at every system boundary.

→ Full rationale in [2.1. ADR - Programming Language](./2.1.%20ADR%20-%20Programming%20Language.md)

---

### 2.3.2 Frontend Framework — Next.js 14 (App Router) + PWA

Next.js provides full-stack TypeScript colocation — frontend pages and API Route Handlers live in the same project, sharing types and eliminating deployment complexity. Server-side rendering (SSR) ensures the public seat availability page delivers real-time data without client-side JavaScript for unauthenticated visitors.

**Progressive Web App (PWA)** is enabled from launch via `next-pwa` to serve mobile users without App Store or Play Store distribution — no app review delays, no 30% platform commission, and no additional talent required beyond our TypeScript team.

→ Full rationale in [2.1. ADR - Programming Language](./2.1.%20ADR%20-%20Programming%20Language.md)

---

### 2.3.3 Database — PostgreSQL + Prisma ORM

PostgreSQL is chosen over MySQL and Oracle for its superior ACID compliance, predictable `SELECT FOR UPDATE` row-level locking (critical for preventing double-booking), first-class `JSONB` support, and zero licensing cost. Oracle is dismissed entirely — commercial per-CPU licensing is incompatible with a startup budget.

Prisma ORM provides a TypeScript-first schema-as-code workflow: the `schema.prisma` file is the single source of truth for database models, TypeScript types, and migration SQL.

→ Full rationale in [2.2. ADR - Database](./2.2.%20ADR%20-%20Database.md)

---

### 2.3.4 Cache — Hazelcast Embedded + AWS Discovery

Hazelcast runs **embedded inside the Node.js process** — no separate cache server is deployed, no additional infrastructure cost at early stage. Seat availability is cached with explicit write-through invalidation on every reservation mutation.

When the platform scales horizontally on AWS, the **`hazelcast-aws` discovery plugin** allows embedded members across all instances to automatically form a cluster — partitioned, replicated, and fault-tolerant — without any application code change. Hazelcast Hot Restart (snapshot persistence) ensures zero data loss on instance replacement.

→ Full rationale in [2.3. ADR - Caching](./2.3.%20ADR%20-%20Caching.md)

---

### 2.3.5 Authentication — Keycloak + Keycloakify

Keycloak is the only candidate that satisfies the **90-day session requirement natively** — configured via `SSO Session Max` in the realm settings, with the refresh token stored in an HTTP-only cookie. Firebase Authentication, the primary alternative, has a hard 14-day cap on session cookies and would require storing refresh tokens in `localStorage` (XSS-accessible) with periodic re-issuance logic.

Keycloak supports our long-term authentication roadmap — Email/Password, Google SSO, and Facebook SSO on Day 1; Phone/OTP, Guest Checkout, Apple SSO, MFA, and SAML/Enterprise SSO via configuration or SPI plugins over time.

**Keycloakify** replaces Keycloak's default Freemarker login templates with custom React + TypeScript + Tailwind components, keeping the login page within our team's skill set and brand design system.

→ Full rationale in [2.4. ADR - Authentication](./2.4.%20ADR%20-%20Authentication.md)

---

### 2.3.6 Payment — AXS Checkout

Self-build is rejected — PCI DSS Level 1 compliance, acquiring bank relationships, fraud detection, and 3D Secure integration require a minimum of 12 months and S$80K–250K/year in audit costs alone.

Between Stripe and AXS Checkout, both satisfy the technical requirements. AXS Checkout is chosen because it is a **Singapore-native, MAS-licensed** company that handles regulatory compliance (Payment Services Act 2019, PCI DSS, AML/KYC), tax obligations (GST invoicing in SGD per IRAS requirements), and data residency (PDPA, Singapore jurisdiction) at the gateway level. Stripe merchants must independently manage these obligations.

AXS Checkout covers all Singapore-relevant payment methods — VISA, Mastercard, Apple Pay, Google Pay, Alipay, WeChat Pay, and PayNow — through a single integration, with full webhook, refund, void, and transaction query API support.

→ Full rationale in [2.5. ADR - Payment](./2.5.%20ADR%20-%20Payment.md)

## 2.4 Key Constraints Driving These Decisions

| Constraint | How the Stack Responds |
|-----------|----------------------|
| TypeScript talent pool | Single language end-to-end — TypeScript in every layer including Keycloakify login theme |
| 90-day session requirement | Keycloak `SSO Session Max` — native, no workaround |
| Zero infrastructure cost at early stage | Hazelcast embedded (no cache server), Firebase rejected (Keycloak self-hosted but replaces identity engineering) |
| Singapore market compliance | AXS Checkout — MAS-licensed, GST/PDPA/IRAS compliance at gateway level |
| Mobile reach without App Store | Next.js PWA — install to home screen, instant updates, no app review |
| Time to market | No self-built auth, payment, or cache infrastructure — all replaced by proven solutions |
| Budget-conscious scaling | PostgreSQL (free), Hazelcast (free), Keycloak (free), AXS (transaction fees only) |
