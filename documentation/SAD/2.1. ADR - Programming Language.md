# 2.1. ADR — Programming Language

**Status:** PROPOSING  
**Date:** 2026-06-16  
**Authors:** HuaHoangLinh

---

## Context

We are a startup building a seat reservation platform in TypeScript. The system must support authenticated users, real-time seat availability, a payment flow, and a 90-day session lifetime. As a startup, our technology choices must balance:

- **Hiring velocity**: We need engineers who can ship across the full stack without deep specialisation silos.
- **Time to market**: Frameworks and tooling must enable fast iteration, not slow us down.
- **Operational simplicity**: We cannot afford large teams dedicated to infrastructure.
- **Scalability headroom**: Choices must not become blockers as we grow.

This document covers the **core platform stack** — language, framework, and UI. Authentication, database, cache, and payment gateway are covered in their own ADRs.

Each decision below is recorded as a standalone ADR entry.

---

## ADR-001 — Language: TypeScript (strict mode)

### Decision
Use TypeScript with `strict: true` as the single programming language across all layers — frontend, backend API, and database schema types.

### Context & Startup Rationale

As a startup, every engineering hire matters. Our **existing talent pool is TypeScript-first** — our engineers are proficient in TypeScript and have built production systems with it. Introducing a second backend language (e.g., Go or Python) would:

- Split our hiring funnel into two pools, making each harder to fill.
- Require engineers to context-switch between languages for frontend and backend work, slowing feature delivery.
- Duplicate shared domain concepts (e.g., `SeatStatus`, `ReservationState`) across language boundaries, creating drift over time.

TypeScript as a **single unified language** means any engineer on our team can pick up any layer of the system on day one.

### Language Comparison

| Language | Type Safety | Full-Stack Capability | Startup Hiring Pool | Our Talent Fit |
|----------|-------------|----------------------|---------------------|---------------|
| **TypeScript** | Strong (compile-time) | Yes — frontend + backend | Large; dominant in web startups | **Primary language** |
| JavaScript | None (runtime errors only) | Yes | Very large | Superset covered by TS |
| Python | Optional (type hints, not enforced) | Backend only (Django/FastAPI); separate frontend needed | Large; strong in data/AI | Not our talent pool |
| Go | Strong (compile-time) | Backend only; separate frontend still needed | Medium; growing but niche | Not our talent pool |
| Java / Kotlin | Strong (compile-time) | Backend only (Spring Boot); heavier framework overhead | Large enterprise pool | Heavyweight for our scale; not our talent pool |
| Rust | Very strong | Backend only; steep learning curve | Small; very niche | Not appropriate for startup velocity |

### Why TypeScript Wins for Us

- **Talent pool alignment**: Our engineers are TypeScript engineers. We hire TypeScript engineers. We do not need to build a separate hiring pipeline for a backend language.
- **Type safety at every boundary**: API request/response shapes, Prisma ORM models, React component props, and shared domain types are all checked at compile time — catching bugs before they reach production.
- **Single source of truth for types**: Domain types defined once in a shared package are used by both the Next.js frontend and backend API routes. No serialisation mismatch between services.
- **Strict mode**: `strict: true` enables the full TypeScript type checker (no implicit `any`, strict null checks, etc.), maximising the safety benefit.
- **Ecosystem**: npm has the largest package ecosystem in the world. Almost every tool, SDK, and library we need has first-class TypeScript support.

### Alternatives Considered

A polyglot architecture (e.g., TypeScript frontend + Go or Python backend) was evaluated and rejected because:
- It fragments our small team across language boundaries.
- It increases onboarding time for new hires.
- It doubles the surface area for dependency management, CI pipelines, and deployment configs.
- Neither Go nor Python provides full-stack capability — a separate frontend framework is still required, which means maintaining two codebases with two separate type systems.

### Consequences
- All npm packages must have adequate `@types/*` coverage or bundled type declarations.
- Strict mode requires more explicit type annotations upfront — accepted as a long-term correctness investment.
- Engineers writing backend API routes must be comfortable with async TypeScript patterns (Promises, `async/await`).

## ADR-002 — Frontend & Backend Framework: Next.js 14 (App Router) + PWA

### Decision
Use Next.js 14 with the App Router as the full-stack framework for both the UI and API layer. Enable Progressive Web App (PWA) capabilities from launch to support mobile users without requiring App Store or Play Store distribution.

### Rationale

#### Full-Stack Colocation
- Frontend pages and API Route Handlers live in the same TypeScript project, eliminating the need to manage two separate repositories or deployments for this scope.
- Shared types flow naturally between the server-side API code and client-side components.

#### SSR for Public Pages
- The seat availability page is server-rendered (React Server Component), so unauthenticated visitors see real-time data on first load — no client-side JavaScript hydration needed.
- Better SEO and Core Web Vitals compared to a pure SPA.

#### React Server Components
- Heavy data-fetching logic runs on the server; only interactive components are sent to the client as JavaScript. Keeps the client bundle lean.

#### PWA — Mobile Expansion Without App Store Review

As a startup, we want to **expand the platform to mobile users quickly**. Shipping a native iOS or Android app introduces:
- **App Store / Play Store review cycles**: Apple's review process can take days to weeks, blocking urgent hotfixes and feature releases.
- **30% platform fee** on in-app purchases.
- **Separate codebase maintenance**: Native apps require Swift/Kotlin expertise outside our TypeScript talent pool.

**PWA solves this entirely**:

| Capability | Native App | PWA (our choice) |
|-----------|-----------|-----------------|
| Install on home screen | Yes | Yes (Android & iOS Safari) |
| Works offline | Yes | Yes (Service Worker cache) |
| Push notifications | Yes | Yes (Web Push API) |
| App Store review required | Yes — days/weeks | **No** |
| 30% platform commission | Yes | **No** |
| Single codebase | No (separate native) | **Yes** |
| Talent required | Swift + Kotlin | **TypeScript (our pool)** |
| Update delivery | Through app store | **Instant — no review** |

By implementing PWA with `next-pwa`, we ship one Next.js codebase that delivers a native-app-like experience on mobile browsers, installs to the user's home screen, and supports offline seat viewing — without going through any app store, without additional hires, and without any platform fee.

### Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| Express + React (separate) | Express provides only low-level routing — SSR, PWA, middleware, and bundling must all be configured manually; two separate projects with duplicated deployment pipelines and no shared type boundary |
| Remix | No built-in PWA support; smaller ecosystem than Next.js with fewer production-proven integrations for the libraries we depend on (Prisma, Keycloak OIDC, AXS) |
| NestJS + Next.js | NestJS is backend-only; pairing it with Next.js requires two separate projects, two deployment pipelines, and type definitions duplicated across the boundary — architectural overhead with no capability gain for this platform |
| React Native | Separate mobile codebase from the web platform; requires native bridge and device API expertise beyond standard TypeScript; does not share components with the web layer |
| Flutter | Dart is outside our TypeScript talent pool; Flutter targets native apps and has no shared codebase with our web platform |
| Expo (React Native) | Managed workflow still requires app store submission for full distribution; web support is secondary and does not deliver full PWA capabilities (service workers, installability) |

### Consequences
- App Router introduces complexity around Server vs Client Components that must be managed carefully.
- PWA requires a valid HTTPS endpoint in production; `localhost` works in development.
- iOS PWA support has some limitations (e.g., push notifications require iOS 16.4+); these are acceptable constraints at this stage.
- Service Worker caching strategy must be designed carefully to avoid serving stale seat availability data offline.

## ADR-003 — Styling: Tailwind CSS

### Decision
Use Tailwind CSS for all UI styling, with a mobile-first approach aligned to the PWA strategy in ADR-002.

### Rationale
- **Speed**: Utility-first classes allow rapid UI construction without writing custom CSS files.
- **Consistency**: Design constraints (spacing, colour, typography) are enforced by the Tailwind config rather than ad-hoc values.
- **Mobile-first**: Tailwind's responsive prefix system (`sm:`, `md:`, `lg:`) makes it natural to design for mobile (PWA) first and scale up to desktop.
- **No runtime cost**: Tailwind is purged at build time; no CSS-in-JS runtime overhead, keeping the PWA payload small.

### Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| CSS Modules | No utility system or responsive breakpoint primitives; every reusable value (spacing, colour) must be defined and maintained manually |
| Styled Components / Emotion | Runtime CSS-in-JS injects styles at render time, increasing JavaScript bundle size and penalising PWA load performance on mobile |
| shadcn/ui + Tailwind | Pre-built component library introduces opinionated structure and accessibility assumptions that conflict with our custom seat reservation UI patterns |

### Consequences
- HTML markup becomes verbose with long class strings. Acceptable at this scale.
- Mobile-first class ordering must be enforced as a team convention.
