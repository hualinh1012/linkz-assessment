# 1.1. Target Architecture

**Status:** VISION  
**Date:** 2026-06-16  
**Authors:** HuaHoangLinh

---

## Purpose

This document describes the target architecture the platform is designed to evolve toward — capable of serving **100K concurrent users (CCU)** and beyond. It is not the current deployed state; it is the destination that every early-stage architectural decision in [1. High-level solution design](./1.%20High-level%20solution%20design.md) is aligned to reach **incrementally and without a full rewrite**.

The transition from the current 3-seat early-stage implementation to this target happens layer by layer as traffic and business demand justify each step.

---

## What Changes at 100K CCU

| Layer | Early Stage (Current) | Target Architecture |
|-------|--------------------|-------------------|
| Reservation model | `SELECT FOR UPDATE` on 3 rows | **Virtual queue** — users enter a queue; reservation processed sequentially |
| Cache | Hazelcast embedded (in-process) | **ElastiCache Redis cluster** — shared distributed cache across all instances |
| Database reads | Single RDS primary | **RDS Proxy + Aurora read replicas** — read/write split, connection pooling |
| Application instances | 2 fixed ECS tasks | **Auto-scaled ECS task fleet** — scales horizontally with demand |
| Keycloak | 2 tasks, shared DB | **Keycloak cluster** — dedicated DB, Infinispan session clustering |
| Payment events | Synchronous webhook handler | **Event-driven** — SQS / EventBridge decouples payment callback from state update |
| Edge protection | ALB only | **CloudFront + WAF + AWS Shield** — DDoS protection, bot filtering, rate limiting |
| Observability | CloudWatch basic | **X-Ray distributed tracing** + CloudWatch Container Insights |

## Key Architectural Shifts Explained

### 1. Virtual Queue — Solving Reservation Contention at Scale

At 100K CCU competing for a limited number of seats, database row locking (`SELECT FOR UPDATE`) creates extreme contention — thousands of requests pile onto the same rows, causing cascading timeouts and poor user experience.

The target architecture replaces direct DB writes with a **SQS FIFO queue**:

```
User requests reservation
        │
        ▼
Enqueue to SQS FIFO
(user gets queue position feedback)
        │
        ▼
Reservation Worker processes queue
one-at-a-time per seat
        │
        ▼
Single DB write per seat — zero contention
```

This decouples user-facing request volume from database write volume entirely. The worker processes at a controlled rate; users receive fair, ordered access to seats.

### 2. ElastiCache Redis — Distributed Cache at Fleet Scale

When the Next.js fleet grows to many ECS tasks, Hazelcast embedded becomes a liability — each task holds its own cache partition, partition rebalancing overhead grows, and the cluster size is bounded by task memory.

ElastiCache Redis provides a **single shared cache** external to the application fleet. Seat availability, rate limit state, and session data are consistent across all Next.js instances. The transition requires only a `CACHE_PROVIDER` environment variable change — the `CacheProvider` abstraction in [2.3. ADR - Caching](./2.3.%20ADR%20-%20Caching.md) handles this with zero application code changes.

### 3. Aurora + RDS Proxy — Read/Write Split and Connection Pooling

At scale, two problems emerge with a single RDS instance:
- **Connection exhaustion**: each ECS task holds open connections; a large fleet exhausts RDS connection limits.
- **Read contention**: seat availability reads compete with reservation writes on the same instance.

RDS Proxy pools connections between the application fleet and Aurora. Aurora read replicas serve all seat availability reads, leaving the Aurora primary exclusively for reservation writes. This is a topology change, not a schema or code change.

### 4. Event-Driven Payment — Decoupling at Scale

At high volume, synchronous AXS webhook processing directly in the API layer risks overloading the application under payment callback spikes. EventBridge decouples the AXS callback receipt from the reservation state update — the callback is acknowledged immediately, and the state machine transition is processed asynchronously by a dedicated consumer.

### 5. WAF + Shield — Edge Protection

At meaningful traffic scale, DDoS attacks and bot traffic become real concerns. AWS WAF rules block malicious patterns (SQL injection, XSS, rate abuse), Shield Standard provides always-on DDoS mitigation, and CloudFront absorbs volumetric attacks at the edge before they reach the application layer.

## Transition Roadmap

Each step below is independently deployable — no big-bang migration required.

| Step | Trigger | Change |
|------|---------|--------|
| **1** | Cache divergence detected across ECS tasks | Hazelcast embedded → ElastiCache Redis (`CACHE_PROVIDER=redis`) |
| **2** | RDS connection limit approaching | Add RDS Proxy in front of existing RDS |
| **3** | Read latency increasing | Add Aurora read replicas; route seat reads to replicas |
| **4** | DB lock contention under high reservation load | Introduce SQS FIFO queue + Reservation Worker |
| **5** | Keycloak becomes auth bottleneck | Separate Keycloak DB; enable Infinispan clustering |
| **6** | DDoS / bot traffic detected | Enable AWS WAF + Shield in front of CloudFront |
| **7** | Payment callback spikes cause API load | Move AXS webhook handling to EventBridge consumer |

Each step is triggered by a measurable signal — not by a calendar date or arbitrary scale threshold. The architecture is designed so that none of these steps require changes to business logic or domain code.
