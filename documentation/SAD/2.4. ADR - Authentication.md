# 2.4. ADR — Authentication

**Status:** ACCEPTED  
**Date:** 2026-06-16  
**Authors:** HuaHoangLinh

---

## Context

Authentication is not a feature we build once and forget. Our long-term product vision requires supporting a range of authentication methods that directly impact user conversion and experience:

- **Email / Password** — standard registration flow
- **Google SSO** — one-tap login, reduces registration drop-off
- **Facebook SSO** — social login for consumer markets

The platform also has an explicit non-functional requirement: **session expiry of 90 days**. This requirement is a first-class constraint that directly influences the identity provider choice.

Building each of these from scratch — token issuance, session management, OAuth2 flows, brute-force protection, token rotation — would consume significant engineering time that should be spent on the core platform. We need an identity solution that natively supports all required methods and meets the session lifetime requirement without workarounds.

---

## Decision

Use **Keycloak** as the Identity Provider (IdP).

Keycloak is self-hosted, open-source, and is the only candidate that satisfies the **90-day session requirement natively** via a single realm configuration — with no application-level workarounds, no client-side token storage risk, and no periodic re-issuance logic required.

## Long-Term Authentication Roadmap

| Method | Keycloak Support | Available From |
|--------|-----------------|---------------|
| Email / Password | Native | Day 1 |
| Google SSO | Native (Social Identity Provider) | Day 1 — configure OAuth2 client in Keycloak admin |
| Facebook SSO | Native (Social Identity Provider) | Day 1 — configure OAuth2 client in Keycloak admin |
| Guest / Anonymous Checkout | Custom Keycloak flow — short-lived token scoped to single reservation session | Long-term |
| Phone / OTP (SMS) | Custom SPI + SMS gateway (e.g., Twilio) | Long-term |
| Apple SSO | Native (Social Identity Provider) | Long-term |
| MFA / TOTP | Native (Google Authenticator compatible) | Long-term |
| SAML / Enterprise SSO | Native | Long-term |

The key principle: **adding or changing an authentication method is a Keycloak configuration or plugin change — not an application code change**. Our Next.js application validates JWTs; it does not care which authentication method produced them.

## Login Page Customisation — Keycloakify

By default, Keycloak renders its own login, registration, and password-reset pages. To maintain a consistent brand experience across the seat reservation platform, we will use **Keycloakify** to replace Keycloak's default UI with fully custom React components.

### What Keycloakify Does

Keycloakify is an open-source library that allows developers to build Keycloak login themes using **React and TypeScript** — the same stack our team already works in. The custom theme is packaged as a `.jar` file and deployed to the Keycloak server, replacing the default Freemarker templates with our React components.

### Why This Matters for Our Team

| Aspect | Default Keycloak UI | Keycloakify |
|--------|-------------------|------------|
| Technology | Freemarker templates (Java-based) | **React + TypeScript** |
| Talent fit | Requires Freemarker / Java knowledge | **Our TypeScript team can own it** |
| Styling | CSS overrides on Keycloak defaults | **Full control — Tailwind CSS** |
| Brand consistency | Separate look-and-feel from the main app | **Matches platform design system** |
| Maintenance | Template duplication, fragile overrides | **Component-based, testable** |

### How It Integrates

```
Keycloakify (React + TypeScript + Tailwind)
        │
        ▼
Build → .jar theme file
        │
        ▼
Deployed to Keycloak /themes directory
        │
        ▼
Keycloak serves our custom React login page
when user is redirected from the Next.js app
```

The Keycloakify theme lives in the same monorepo as the Next.js application — sharing the design system, Tailwind config, and component patterns with zero context switching for the development team.

## The 90-Day Session — Why This Is the Deciding Factor

The 90-day session requirement is an explicit functional requirement. The comparison below shows why this makes Keycloak the only clean solution.

### Keycloak — Native, Secure, Zero Code

```
Keycloak Admin Console
  → Realm Settings → Tokens
    → SSO Session Max: 90 days   ✅ done
```

- Refresh token lifetime is set to 90 days at the IdP level.
- Tokens are stored in HTTP-only cookies — never accessible to JavaScript.
- Standard OIDC middleware handles silent refresh transparently.
- No application business logic required.

### Firebase — Cannot Meet This Requirement Natively

Firebase `createSessionCookie()` has a **hard maximum of 14 days**:

```typescript
// ❌ This throws a runtime error — Firebase rejects anything over 14 days
const expiresIn = 90 * 24 * 60 * 60 * 1000;
await adminAuth.createSessionCookie(idToken, { expiresIn });
```

To simulate 90 days with Firebase, the application must:
1. Issue a 14-day session cookie.
2. Store Firebase's refresh token in `localStorage` or `IndexedDB` (readable by JavaScript).
3. Implement background re-issuance logic to silently exchange the refresh token for a new ID token and re-issue a new 14-day session cookie every ~13 days.
4. Handle race conditions, failure cases, and tab-sync edge cases for the re-issuance.

| Risk | Impact |
|------|--------|
| Refresh token in `localStorage` | Accessible to JavaScript → XSS can steal it |
| Re-issuance failure | User is unexpectedly logged out mid-session |
| Background logic complexity | Non-trivial engineering effort for a requirement that Keycloak solves with one config line |

**Firebase is cheaper at early stage and has better guest checkout UX — but it cannot meet the 90-day session requirement safely without significant workaround complexity.**

## Why Other Candidates Were Rejected

### Self-Build
Building auth from scratch means implementing: password hashing, JWT issuance and rotation, OAuth2 client flows per social provider, brute-force throttling, session revocation, and OTP delivery. Estimated 6–10 weeks — entirely avoidable.

### Firebase Authentication
Strong free tier and native guest checkout. Rejected because `createSessionCookie()` is capped at 14 days — meeting the 90-day session requirement requires storing refresh tokens in `localStorage` (XSS exposure) and implementing background cookie re-issuance logic. This trades a free tier advantage for a security and complexity cost that is not acceptable.

### AWS Cognito
Comparable free tier and configurable session lifetime. Rejected because:
- Guest/anonymous auth is complex — requires Cognito Identity Pools, a separate service from Cognito User Pools, with a significantly more involved integration.
- Developer experience is consistently poor — SDK verbosity and console UX are known pain points.
- AWS vendor lock-in propagates across every service that needs to verify tokens.

### Auth0
Best developer experience in the category. Rejected on cost — $1,400/month at 100K MAU is incompatible with startup budget at any meaningful scale beyond the free tier.

### Clerk
Strong DX with pre-built React components. Rejected because pre-built components conflict with our custom seat reservation UI and PWA design requirements. No self-hosting option. Cost scales unfavourably with MAU growth.

## How It Fits — Integration with Next.js

```
User (Browser)
      │
      ├── Email/Password ──► Keycloak login page (hosted)
      ├── Google SSO ───────► Keycloak → Google OAuth2 → Keycloak token
      ├── Facebook SSO ─────► Keycloak → Facebook OAuth2 → Keycloak token
      └── Guest ────────────► Keycloak custom flow → short-lived scoped token
                                        │
                              OIDC Authorization Code Flow
                                        │
                                        ▼
                              Next.js middleware receives
                              access_token + refresh_token
                              stored in HTTP-only cookies
                                        │
                                        ▼
                              Next.js middleware verifies
                              JWT signature via Keycloak JWKS endpoint
                              on every protected request
                              (silent refresh when access token expires)
```

The Next.js application never handles passwords or OAuth tokens directly. It only receives and verifies Keycloak-issued JWTs.

## Session Configuration

| Parameter | Value | Where Configured |
|-----------|-------|-----------------|
| Access token lifetime | 15 minutes | Keycloak realm → Tokens |
| Refresh token lifetime (SSO Session Max) | **90 days** | Keycloak realm → Tokens → SSO Session Max |
| Token storage | HTTP-only cookies (XSS-safe) | Next.js middleware |
| Token verification | Keycloak JWKS endpoint (RS256) | Next.js middleware |
| Silent refresh | Next.js middleware refreshes access token using refresh token before expiry | Application middleware |
| Session revocation | Keycloak admin API or user-initiated logout | Keycloak + Next.js logout route |
| Guest token lifetime | 2 hours | Keycloak custom flow scope |

## Consequences

- Keycloak requires a self-hosted deployment — a Docker container locally and a managed container service (ECS, Cloud Run) in production (~$15–20/month). This is one additional service to operate but replaces the entire identity engineering workload.
- Phone/OTP and Guest Checkout require Keycloak SPI plugins — moderate development tasks (1–2 weeks each) but one-time investments reusable across all future products on the same realm.
- All protected routes verify the Keycloak JWT in Next.js middleware using Keycloak's JWKS endpoint. Keycloak must be treated as a high-availability service — downtime makes protected routes inaccessible.
- Social providers (Google, Facebook) require OAuth2 app credentials registered in their respective developer consoles.
- The `keycloak_id` (`sub` claim from the JWT) is stored in the local `users` table as a unique external identity reference (not a SQL foreign key — there is no local table it points to) — Keycloak is the source of truth for identity; PostgreSQL holds only application-specific data.
