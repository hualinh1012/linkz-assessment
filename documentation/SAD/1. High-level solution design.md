# 1. High-Level Solution Design

## 1.1 Purpose

This document describes the high-level solution design for the **Linkz Seat Reservation Platform** — a small, public-facing web application that allows authenticated users to view, reserve, and cancel seats, backed by AXS Checkout for real payment processing.

---

## 1.2 Problem Statement

We need a minimal seat reservation system where:

- Any visitor can view the availability of 3 seats.
- Only authenticated users may reserve a seat.
- Reserving a seat requires completing a payment step.
- Sessions must expire after 90 days.

---

## 1.3 Goals & Non-Goals

### Goals
- Allow public visibility of seat availability (no login required to view).
- Secure seat reservation behind authentication.
- Enforce a 90-day session lifetime.
- Guide users through a seat selection → payment → confirmation flow.
- Prevent double-booking (a reserved seat cannot be taken by another user).
- Allow users to cancel a confirmed reservation with an AXS refund and seat release.

### Non-Goals
- Multi-venue or dynamic seat count at this stage (fixed at 3 seats — see Assumption #11 for long-term intent).
- Admin panel or seat management UI.
- Email/SMS notifications.
- Native iOS / Android apps (mobile is served via PWA).

---

## 1.4 Stakeholders

| Role | Concern |
|------|---------|
| End User (Guest) | Can view seat availability without logging in |
| End User (Authenticated) | Can reserve a seat after logging in and completing payment |
| Developer | Simple, maintainable TypeScript codebase |
| Assessor | Clean architecture, correct flow, production-ready patterns |

---

## 1.5 Assumptions & Constraints

> **Key Architectural Constraint**
>
> The current requirement is **3 fixed seats**. However, this platform is intentionally designed and built for large-scale growth. Every technology choice, data model decision, and infrastructure pattern is made with a clear, incremental transition path from this early-stage implementation to a high-volume, multi-venue seat reservation system — **without requiring a full rewrite**. Each layer has a defined evolution path documented in the respective ADRs under [2. Technology & Integration Adoption](./2.%20Technology%20%26%20Integration%20Adoption.md).

| # | Assumption / Constraint |
|---|------------------------|
| 1 | The system manages exactly 3 seats at this stage — the data model and architecture are seat-count agnostic and will support dynamic seat inventory as the platform grows. |
| 2 | Payment is handled by AXS Checkout (Payment Page) — a redirect-based hosted checkout by AXS. |
| 3 | A seat becomes reserved only after payment is confirmed — not at selection time. |
| 4 | A user may hold at most one active reservation at a time — enforced at the database level. |
| 5 | A `PENDING` reservation expires after **15 minutes** if payment is not completed; the seat is automatically released back to `AVAILABLE`. This aligns with AXS Checkout's maximum payment page session duration. |
| 6 | Confirmed reservations can be cancelled by the user; cancellation triggers an AXS refund (void same-day, refund post-settlement) and releases the seat. |
| 7 | Session expiry is 90 days from the time of login. |
| 8 | The stack must be TypeScript end-to-end. |
| 9 | Keycloak is self-hosted on AWS ECS (same deployment as the application). |
| 10 | AXS Checkout provides a sandbox environment for development and testing — no live transactions required during development. |
| 11 | See key architectural constraint above — small start, large-scale vision, clear transition roadmap. |

   │                    │                     │

   │             Redirect to AXS        Seat released

   │             (PayNow / wallets / cards)

   │             │             │
   │        Payment OK    Payment Failed / Cancelled

   │             │

   │      ┌──────┴────────────┐

## 1.7 Proposed Technology Stack

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| Frontend | Next.js 14 (App Router) + React | Full-stack TypeScript, SSR for public seat view |
| Styling | Tailwind CSS | Rapid, utility-first UI |
| Backend | Next.js API Routes (Route Handlers) | Co-located with frontend, avoids separate server |
| Database | PostgreSQL via Prisma ORM | Production-grade relational DB with permanent persistence |
| Authentication | Keycloak | Self-hosted identity provider; native 90-day session via `SSO Session Max` config; supports email/password, Google SSO, Facebook SSO, and extensible to phone/OTP and guest checkout via SPI; issues standard OIDC JWTs |
| Cache | None at this stage | Seat-availability reads go directly to PostgreSQL — at 3 seats the read is trivial and a cache adds cost/complexity for no benefit. A Redis cache is introduced only when volume justifies it (see [2.3. ADR - Caching](./2.3.%20ADR%20-%20Caching.md)) |
| Payment | AXS Checkout (Payment Page) | Singapore-based online payment gateway; low-code redirect-based checkout page hosted by AXS; supports PayNow, digital wallets, and cards without the app handling sensitive card data |
| Language | TypeScript (strict mode) | End-to-end type safety |

## 1.9 Key Design Decisions

| Decision | Choice | Reason |
|----------|--------|--------|
| Identity provider | Keycloak | Decouples auth logic from application; only candidate that meets 90-day session natively; supports multiple login methods with minimal app-level changes; issues standard OIDC JWTs |
| Session lifetime | 90-day refresh token via Keycloak `SSO Session Max` config | Single realm setting — no application code or workaround required; refresh token stored in HTTP-only cookie |
| JWT validation | Next.js middleware verifies Keycloak-issued JWT via JWKS endpoint on each request | Stateless; no session table needed in the application DB |
| User identity in DB | `users` table stores `keycloak_id` (`sub` claim) as a unique external identity reference (not a SQL foreign key) | Keycloak is source of truth for identity; PostgreSQL holds application-specific data only |
| Database | PostgreSQL | Permanent persistence, ACID transactions, suitable for concurrent reservation writes |
| Seat availability cache | None at this stage | At 3 seats the availability read is a sub-millisecond indexed query; a cache optimises a non-bottleneck and adds cost/operational surface. Reads go straight to PostgreSQL (see [2.3. ADR - Caching](./2.3.%20ADR%20-%20Caching.md)) |
| Cache invalidation | N/A | No cache means no invalidation to maintain — PostgreSQL is the single source of truth and every read is fresh by construction |
| Payment gateway | AXS Checkout (Payment Page) | User is redirected to AXS-hosted checkout page; AXS handles all payment UI and card data; app receives result via return URL redirect; eliminates PCI DSS scope for card data |
| Seat reservation lifecycle | `PENDING` (15-min expiry) → AXS Payment Page → `RESERVED` on success → `AVAILABLE` on failure, expiry, or cancellation | Prevents double-booking; 15-minute expiry releases stale PENDING seats automatically |
| Cancellation & refund | `DELETE /api/reservations/:id` triggers AXS void (same-day) or refund (post-settlement) and releases seat | Cancellation is user-initiated; AXS payment reversal type depends on settlement status |
| One reservation per user | Unique constraint on active reservation per user — a second reservation attempt is rejected if one is PENDING or RESERVED | Prevents a user from holding multiple seats; enforced at DB level |
| Public seat view | SSR (Next.js Server Component) | Real-time availability for unauthenticated visitors without relying on client-side fetches |
| Monorepo vs split | Single Next.js project | Appropriate for scope; avoids unnecessary complexity |

## 1.10 Resolved Questions

| # | Question | Answer | Impact |
|---|----------|--------|--------|
| Q1 | Can a user reserve more than one seat? | **No** — one active reservation per user at a time | Unique constraint on user + active status enforced at DB level; second attempt rejected |
| Q2 | Can a reservation be cancelled after payment? | **Yes** | Cancellation flow added: AXS void (same-day) or refund (post-settlement); seat released to `AVAILABLE` |
| Q3 | How long should a `PENDING` reservation be held before expiry? | **15 minutes** — capped by AXS Checkout's maximum payment page session duration | Background job or DB-level TTL releases stale `PENDING` seats after 15 minutes; aligns with AXS session expiry so no user can be on a valid payment page while the reservation has already expired |
| Q4 | Which Keycloak realm and client will be used? Self-hosted or managed? | **Self-hosted on AWS ECS** | Keycloak runs as an ECS Fargate service in the same AWS VPC; realm and client configuration managed via Keycloak admin console |
| Q5 | Are AXS Checkout credentials available for development? | **Yes — AXS provides a sandbox environment** | Development and staging integrate against AXS sandbox; no live transactions during development |
