# 2.3. ADR — Caching

**Status:** ACCEPTED  
**Date:** 2026-06-17  
**Authors:** HuaHoangLinh

> **Supersedes** the previous proposal to use Hazelcast embedded mode. See [Decision](#decision) for why that was reversed.

---

## Context

Seat availability is the most frequently read data in the platform — queried on every page load by authenticated and unauthenticated users alike. The original proposal was to put this behind **Hazelcast in embedded mode**, on the premise that it would give us an in-process, distributed-capable cache with *zero* additional infrastructure.

Two facts invalidated that premise:

1. **Hazelcast embedded mode is JVM-only — it is not compatible with our Node.js/TypeScript runtime.** "Embedded" means the cache member runs *inside* the application process; that only works on the JVM. On Node.js the only option is the Hazelcast **client**, which is a thin client that still requires a **separately deployed JVM Hazelcast cluster**. The zero-infrastructure advantage — the entire reason embedded mode was chosen — does not exist for us. It collapses into exactly the standalone-cache cost we were trying to avoid.

2. **At this stage the volume does not justify any cache at all.** The platform has **3 seats**. Seat availability is a single indexed primary-key read over a handful of rows, which PostgreSQL serves in well under a millisecond. Caching it optimises a non-bottleneck.

---

## Decision

**Do not introduce any caching layer at this stage.** Seat-availability reads are served **directly from PostgreSQL** through the repository layer.

This is driven by **volume and budget**:

- **Volume** — 3 seats and a small user base. The read this cache would serve is already trivial for PostgreSQL; a cache would add complexity to optimise something that is not slow.
- **Budget** — the only Node-compatible path is a *separately deployed* cache (a JVM Hazelcast cluster, or a managed store like ElastiCache). Either adds ongoing cost and operational surface (monitoring, failover, eviction tuning) for no measurable benefit at this scale.
- **Correctness simplicity** — with no cache there is no invalidation to get wrong. PostgreSQL is the single source of truth and every read is fresh by construction. This removes an entire class of staleness bugs.

---

## Consequences

- `GET /api/public/seats` and the SSR seat page read directly from PostgreSQL (see [TSD/2.2. List of Seats](../TSD/2.2.%20List%20of%20Seats.md)).
- **No cache invalidation/eviction logic exists anywhere** — the reservation, payment, cancellation, and expiry-cleanup flows no longer carry an "evict cache" step.
- **Rate-limit state** (previously slated for the cache) is held **in-process per instance** at this stage — best-effort across the 2 ECS tasks, which is acceptable for the current volume. It moves to a shared store when caching is introduced (see below and [TSD/1.4. API Specification §2.7](../TSD/1.4.%20API%20Specification.md)).
- There is no `CACHE_PROVIDER` / `REDIS_URL` configuration in the current environment (see [TSD/1.7. Environment & Configuration](../TSD/1.7.%20Environment%20%26%20Configuration.md)).

---

## Evolution Path — When a Cache *Is* Justified

The target architecture still includes a distributed cache; we are simply not paying for it before it earns its keep. The trigger to introduce one is **measurable**, not calendar-based:

- Seat-availability read latency or PostgreSQL read load rises materially, **or**
- The application fleet grows beyond a couple of instances and needs **shared** state (rate limiting, hot reads) across them.

When that happens, introduce **AWS ElastiCache for Redis** behind a `CacheProvider` port. Redis is the choice (not Hazelcast): it is a first-class fit for Node.js with a mature client, it is fully managed on AWS, and any RESP-compatible store (Valkey, KeyDB, Dragonfly) is a drop-in substitute.

| Candidate | Node.js fit | Managed on AWS | Notes |
|-----------|-------------|----------------|-------|
| **Redis (ElastiCache)** | First-class | Yes | Chosen target; mature `ioredis`/`node-redis` clients |
| Valkey / KeyDB / Dragonfly | First-class (RESP) | Partial / self-host | Drop-in Redis-protocol substitutes; no app change |
| Hazelcast | Poor (JVM-embedded; Node is thin-client only) | No | Rejected — see Context |
| Memcached | OK | Yes (ElastiCache) | No data structures / persistence; weaker fit |

### The `CacheProvider` seam (deferred, not built yet)

Because of the [Clean Architecture](../TSD/1.1.%20Project%20Architecture.md) boundaries, adding the cache later is additive — no domain or use-case logic changes. The seam is a port introduced **at that time**:

```typescript
// application/ports/cache-provider.ts  (introduced when Redis is added)
interface CacheProvider {
  get<T>(key: string): Promise<T | null>;
  set<T>(key: string, value: T, ttlSeconds?: number): Promise<void>;
  evict(key: string): Promise<void>;
}
```

Introducing it is: add the port, add a `RedisCacheProvider` adapter, have the `ListSeats` use case consult it (cache-aside with write-through eviction on reservation mutations), and wire it via `CACHE_PROVIDER=redis` + `REDIS_URL`. Until then, `ListSeats` depends only on the seat repository.

See [SAD/1.1. Target Architecture](./1.1.%20Target%20Architecture.md) for where this sits in the broader scaling roadmap.

---

## Why Not Cache "Just in Case"

Adding a cache now would mean: standing up and paying for a Redis instance (or a JVM cluster), writing and code-reviewing invalidation on every mutation, and carrying the staleness-bug risk — all to speed up a sub-millisecond primary-key read on 3 rows. That is architectural overhead with no payoff at this scale. The lean choice is to read PostgreSQL directly and add the cache the moment, but not before, a metric says it is needed.
