# 2.3. ADR — Caching

**Status:** PROPOSING  
**Date:** 2026-06-16  
**Authors:** HuaHoangLinh

---

## Context

Seat availability is the most frequently read data in the platform — queried on every page load by both authenticated and unauthenticated users. Without caching, every request hits PostgreSQL for a value that changes infrequently (only on reservation write or cancellation). At early stage this is manageable, but the pattern does not scale and introduces unnecessary database cost.

We need a caching layer that:

- Reduces redundant PostgreSQL reads for seat availability.
- Is consistent — a reserved seat must never appear available from cache.
- Has a clear invalidation strategy tied to reservation state changes.
- **Does not require a separately deployed infrastructure component** at early stage, when operational budget and DevOps capacity are limited.
- Can evolve to a distributed topology as the platform scales, **without requiring application code changes**.

---

## Decision

Use **Hazelcast in embedded mode** as the caching layer. Hazelcast embedded is not just an early-stage shortcut — it is a topology that scales from a single instance to a fully distributed, highly available cluster on AWS through member discovery, without deploying any separate cache infrastructure and without changing application code.

---

## Why Embedded Mode Wins at Early Stage

Deploying a standalone distributed cache (Redis, Memcached, etc.) at early stage introduces costs that outweigh the benefit:

| Concern | Standalone Cache (Redis, etc.) | Hazelcast Embedded |
|---------|-------------------------------|-------------------|
| Infrastructure cost | Additional server or managed service cost (e.g., AWS ElastiCache starts at ~$15–30/month even for dev) | **Zero** — runs inside the existing Node.js process |
| Deployment complexity | Separate container, health checks, connection config, failover | **None** — part of the application deployment |
| Operational overhead | Requires monitoring, memory tuning, and eviction policy management as a standalone service | Managed inside the application process lifecycle |
| Network latency | Cross-process TCP call (~0.1–1ms per cache hit) | **Sub-millisecond** — in-process memory access |
| Early-stage DevOps capacity | Requires dedicated attention to keep the cache service healthy | **None** — lives and dies with the application process |

For a startup with 3 seats and a small user base, **bleeding budget on a standalone cache service is an architectural overhead that delivers no measurable benefit at this scale**. Hazelcast embedded gives us a production-grade caching API with zero infrastructure cost.

## Hazelcast AWS Mode — High Availability at Scale

When the platform scales horizontally on AWS (multiple application instances behind a load balancer), Hazelcast embedded members do not remain isolated. Using the **`hazelcast-aws` discovery plugin**, each embedded Hazelcast member automatically discovers other members running in the same AWS environment and forms a cluster — without deploying any separate cache server.

### How Member Discovery Works on AWS

```
AWS ECS / EC2 Auto Scaling Group
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  ┌──────────────────┐     ┌──────────────────┐          │
│  │  App Instance A  │     │  App Instance B  │          │
│  │                  │     │                  │          │
│  │  Next.js         │     │  Next.js         │          │
│  │  + Hazelcast     │◄───►│  + Hazelcast     │          │
│  │    Member        │     │    Member        │          │
│  │  [Partitions     │     │  [Partitions     │          │
│  │   0–134 + backup]│     │   135–270 + backup]         │
│  └──────────────────┘     └──────────────────┘          │
│           ▲                        ▲                    │
│           └──── AWS tag discovery ─┘                    │
│                 (hazelcast-aws plugin)                   │
└─────────────────────────────────────────────────────────┘
```

Hazelcast AWS discovery uses EC2 tags or ECS task metadata to locate peers. When a new instance starts, it announces itself; the cluster automatically rebalances partitions across all live members.

### Data Partitioning and Replication

Hazelcast divides all cached data into **271 partitions** distributed evenly across cluster members. Each partition has a configurable number of **backup copies** stored on different members (and optionally different Availability Zones):

| Partition state | Behaviour |
|----------------|-----------|
| Primary partition | Serves reads and writes for its key range |
| Backup partition | Holds a synchronised copy on a different member |
| Member failure | Backup partition is promoted to primary automatically on surviving members |

This means **no data is lost when an instance goes down**. The remaining members already hold backup copies and continue serving requests without interruption.

### Snapshot and Persistence on Instance Failure

Hazelcast's **Hot Restart (Persistence)** feature allows each member to take periodic snapshots of its partition data to disk. When an instance is recycled or replaced (e.g., ECS task replacement, EC2 spot interruption):

1. The new instance starts and joins the cluster via AWS discovery.
2. It loads its persisted snapshot from disk (or EFS-mounted volume).
3. It synchronises with the cluster to receive any changes that occurred while it was down.
4. The cluster is fully restored — **zero data loss, zero manual intervention**.

### Non-Functional Requirements Addressed

| NFR | How Hazelcast AWS Mode Satisfies It |
|-----|-------------------------------------|
| **High Availability** | Backup partitions on separate members / AZs; automatic failover on instance failure; no single point of failure in the cache layer |
| **Performance** | In-process reads for locally-owned partitions (sub-millisecond); no cross-process TCP hop for the majority of cache hits; Near Cache option available for ultra-hot keys |
| **Zero Bottleneck** | No centralised cache server that all instances funnel through; each application instance is itself a cache member, distributing load linearly as instances scale |

## Distributed Cache Technologies — Comparison

When the platform grows and embedded mode is no longer sufficient (e.g., multiple application instances need a shared cache), the following technologies are candidates:

| Technology | Type | Protocol | Persistence | Startup Suitability | Notes |
|-----------|------|----------|-------------|--------------------|-|
| **Hazelcast Cluster** | Distributed in-memory | Hazelcast protocol | Optional | High — same API as embedded; zero code change | Natural upgrade path from embedded mode |
| **Redis** | Standalone / Cluster | RESP | Optional (RDB/AOF) | High — largest ecosystem, best tooling | Industry standard; Hazelcast client can be swapped in |
| **Valkey** | Standalone / Cluster | RESP-compatible | Optional | High — open-source Redis fork post-license change | Drop-in Redis replacement; avoids Redis Ltd. licensing risk |
| **KeyDB** | Standalone / Cluster | RESP-compatible | Optional | Medium | Multi-threaded Redis fork; higher throughput per node |
| **Dragonfly** | Standalone | RESP-compatible | Optional | Medium | Modern Redis-compatible; high throughput, low memory footprint |
| **Memcached** | Standalone / Cluster | Memcached protocol | None | Medium — simple key-value only | No data structures beyond strings; no pub/sub or sorted sets |
| **Apache Ignite** | Distributed in-memory | Native / SQL | Optional | Low — heavy JVM dependency | Strong for compute grid use cases; heavyweight for pure caching |
| **Infinispan** | Distributed in-memory | Hot Rod / REST | Optional | Low — Red Hat ecosystem | Enterprise-grade but complex setup; better fit for Java stacks |

**Key insight**: All RESP-compatible alternatives (Valkey, KeyDB, Dragonfly) are interchangeable with Redis at the protocol level. If we move to Redis-compatible distributed cache at scale, the application code does not change — only the connection configuration.

## What Is Cached

| Cache Key | Value | TTL | Invalidation Trigger |
|-----------|-------|-----|---------------------|
| `seats:all` | Array of all seat records with current status | 60 s (safety TTL) | On any reservation creation, update, or cancellation |

The 60-second safety TTL acts as a backstop against cache poisoning in edge cases where explicit invalidation is missed. Under normal operation, invalidation is always explicit and immediate.

---

## Cache Abstraction — Adapter to Distributed Mode

The application code will never call Hazelcast directly. All cache operations go through a `CacheProvider` interface:

```typescript
interface CacheProvider {
  get<T>(key: string): Promise<T | null>;
  set<T>(key: string, value: T, ttlSeconds?: number): Promise<void>;
  evict(key: string): Promise<void>;
}
```

Two implementations will be provided:

```
CacheProvider (interface)
    ├── HazelcastEmbeddedCacheProvider   ← active now (embedded mode)
    └── RedisCacheProvider               ← ready for distributed scale
```

Switching from Hazelcast embedded to Redis (or any RESP-compatible store) requires only:
1. Setting an environment variable `CACHE_PROVIDER=redis`
2. Providing `REDIS_URL` in the environment config

**No application business logic changes.** The `SeatsService`, `ReservationService`, and all callers remain unchanged.

---

## Invalidation Strategy

Cache invalidation is **explicit and write-through** — the cache is never allowed to drift from the database:

```
User reserves seat
        │
        ▼
PostgreSQL transaction commits (seat status → PENDING / RESERVED)
        │
        ▼
CacheProvider.evict('seats:all')
        │
        ▼
Next read → cache miss → re-populate from PostgreSQL → cache set
```

This guarantees that no user ever sees a stale seat status after a reservation event. The eviction call is always made within the same request handler that commits the database write — it is not deferred or background-scheduled.

## Consequences

- At single-instance deployment (early stage), Hazelcast runs fully in-process with zero infrastructure cost or operational overhead.
- At multi-instance deployment on AWS, enabling the `hazelcast-aws` discovery plugin is the only configuration change required — application code, business logic, and the `CacheProvider` interface remain untouched. Instances automatically form a cluster and share partitioned, replicated cache data.
- Hazelcast Hot Restart must be configured with a persistent volume (e.g., EFS mount on ECS) to enable snapshot recovery after instance replacement. This is an infrastructure configuration task, not a code change.
- If a standalone cache is ever required for a language-agnostic, multi-service architecture (e.g., a Python data service also needs access to the same cache), the `CacheProvider` adapter allows switching to Redis or Valkey via environment variable with no code changes.
- Cache invalidation must be called explicitly on every reservation mutation. This is a **code discipline requirement** enforced in code review — a missed eviction call results in stale availability data being served across all cluster members.
- The 60-second safety TTL means a worst-case stale window of 60 seconds only occurs if explicit invalidation fails — which should never happen under normal operation.
