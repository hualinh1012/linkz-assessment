**Status:** ACCEPTED

## Context

The seat reservation platform requires a payment step to complete a reservation. A user selects a seat, proceeds to payment, and the seat is only confirmed as reserved upon successful payment. This means the payment layer is directly coupled to the core business flow — a failed, delayed, or insecure payment means a broken reservation.

Key requirements for the payment solution:

- **Singapore market**: Must support payment methods commonly used in Singapore (PayNow, VISA, Mastercard, digital wallets).
- **Security**: Card data must never touch our application servers — PCI DSS scope must be minimised.
- **Reliability**: Payment confirmation must be delivered reliably to drive the seat reservation state transition.
- **Refund / void support**: If a reservation is cancelled, the payment must be refundable or voidable via API.
- **Time to market**: The solution must be integrable within days, not months.
- **SME-friendly**: As a startup, we need reasonable pricing without minimum volume commitments.

---

## Decision

Use **AXS Checkout** as the payment gateway, integrated via the **Payment Page** (redirect-based hosted checkout) with **webhook** delivery for payment confirmation.

## Why Self-Build Is Rejected

Building a payment system in-house is consistently the most underestimated engineering challenge in fintech. It is rejected without further evaluation for the following reasons:

### 1. PCI DSS Compliance — Minimum 12 Months, Significant Cost
To process card payments directly, the platform must achieve **PCI DSS Level 1** compliance — the highest tier, required for any system that stores, processes, or transmits cardholder data. This involves:

- Engagement of a **Qualified Security Assessor (QSA)** for an annual on-site audit.
- Quarterly network scans by an **Approved Scanning Vendor (ASV)**.
- Annual penetration testing.
- Extensive documentation: network diagrams, data flow maps, security policies, incident response plans.
- Estimated cost: **S$80,000–S$250,000 per year** in audit and remediation fees alone.

No startup should absorb this cost when PCI-compliant gateways exist at transaction-fee-only pricing.

### 2. Acquiring Bank Relationships
Processing card payments requires a direct relationship with an acquiring bank or payment processor. Establishing this relationship — KYC, underwriting, settlement agreements — takes **3–6 months** for a new business with no transaction history.

### 3. Fraud Detection, 3D Secure, Chargebacks
A production payment system requires: real-time fraud scoring, 3D Secure (SCA) integration per card scheme rules, and chargeback management infrastructure. Each of these is a non-trivial subsystem.

### 4. Does Not Meet Time to Market
Combined, building and certifying a payment system takes a **minimum of 12 months** for a well-resourced team. This is incompatible with our time-to-market requirement.

## Why AXS Checkout Over Stripe

Both Stripe and AXS Checkout are credible, production-grade payment gateways that satisfy the technical requirements. The decision is AXS Checkout for the following reasons:

### 1. Singapore-Native — Compliance, Tax, and Support Out of the Box

AXS has operated in Singapore since 2000 and is licensed by the **Monetary Authority of Singapore (MAS)** under the Payment Services Act 2019. Being a Singapore-incorporated company, AXS carries compliance obligations that directly benefit merchants who integrate with it — removing regulatory and administrative burden that would otherwise fall on our team.

| **PCI DSS** | AXS maintains PCI DSS certification on its infrastructure; using the hosted Payment Page keeps our platform entirely out of PCI scope for card data |
| **3D Secure / SCA** | AXS enforces Strong Customer Authentication aligned with MAS expectations for Singapore card transactions |
| **AML / KYC** | AXS performs Anti-Money Laundering checks as part of its MAS obligations — merchant onboarding is covered |

AXS Checkout explicitly targets **SMEs and new businesses** with accessible onboarding (no ACRA registration required, MyInfo verification available) and pricing designed for lower-volume merchants. Stripe's self-serve model has no equivalent SME programme in Singapore.

### 5. Complete Integration Capabilities

AXS Checkout provides everything needed for the reservation payment lifecycle:

| Capability | How We Use It |
|-----------|--------------|
| **Payment Link (hosted)** | User is redirected to AXS-hosted checkout; card data never touches our servers; PCI scope is minimal |
| **Webhook** | AXS posts JWE-encrypted payment result to `/api/public/payment/callback`; drives all DB state transitions (PENDING → RESERVED/CANCELLED) |
| **successUrl / failUrl** | Browser redirect targets for user navigation after payment completes; DB state is not updated from these — the webhook is authoritative |
| **Void** | Attempt first when cancelling a confirmed reservation; reverses an unsettled payment |
| **Refund API** | Fallback when void is rejected (payment already settled); issues a full refund |
| **Card on File** | Future: returning users can pay without re-entering card details |

## Payment Flow

```
User selects seat
        │
        ▼
POST /api/reservations
  → Create PENDING reservation in DB (SELECT FOR UPDATE)
  → Encrypt JWE payment link (clientId, amount, merchantRef,
    webhookUrl, successUrl, failUrl)
  → Return redirectUrl to client
        │
        ▼
Browser redirects to AXS-hosted Payment Page
  (VISA / Mastercard / Apple Pay / Google Pay /
   Alipay / WeChat Pay / PayNow)
        │
   ┌────┴────┐
   │         │
Payment    Payment
Success    Failed / Expired
   │         │
   ▼         ▼
AXS redirects browser to successUrl or failUrl
  (user navigation only — no DB change at this point)
        │
        ▼
AXS → POST /api/public/payment/callback  (JWE-encrypted webhook)
  → Decrypt JWE with AXS_API_SECRET
  → Idempotency check
  │
  ├── SUCCESS  → UPDATE seat → RESERVED
  │              UPDATE reservation → CONFIRMED
  │              UPDATE payment → SUCCESS
  │
  └── DECLINED / EXPIRED → UPDATE seat → AVAILABLE
                           UPDATE reservation → CANCELLED
                           UPDATE payment → FAILED / EXPIRED
```

**AXS redirects the user first, then delivers the webhook.** The `successUrl`/`failUrl` pages are for user experience only; the webhook is the authoritative source that drives all DB state transitions. The seat state is always driven by a verified, server-side AXS response — never by client-side redirect parameters.

## Consequences

- AXS merchant credentials (`AXS_CLIENT_ID` and `AXS_API_SECRET`) must be provisioned before development begins. A sandbox/test environment is available via the AXS developer portal (`developer.svc.axsasia.com`) for integration testing without real transactions.
- Webhook endpoint (`/api/public/payment/callback`) must be publicly reachable during development. A tunnelling tool (e.g., ngrok) is required for local testing.
- Webhook payloads must be verified using the AXS-provided signature before processing — unverified callbacks must be rejected to prevent payment spoofing.
- AXS Checkout's payment page has a **maximum session duration of 15 minutes**. The application's `PENDING` reservation expiry is set to match this cap — ensuring a seat is never held longer than the payment window AXS permits. Any user still on an AXS payment page when the reservation expires will find the seat released upon returning.
- Void vs refund distinction must be handled in the cancellation flow: void applies before settlement (same day), refund applies after settlement.
- AXS Payment Page aesthetics are partially controlled by AXS branding. Custom logo and colour configuration is available within AXS's theming options.
