# 2.2. ADR — Database

**Status:** ACCEPTED  
**Date:** 2026-06-16  
**Authors:** HuaHoangLinh

---

## Context

The seat reservation platform requires a relational database to persist users, seats, and reservations. The data model is relational by nature — a reservation links a user to a seat, seat status must be consistent across concurrent requests, and the payment lifecycle introduces state transitions that must be atomic.

Key requirements that drive the database choice:

- **Permanent persistence**: Data must survive application restarts and container recycling.
- **Concurrent write safety**: Two users must not be able to reserve the same seat simultaneously.
- **ACID transactions**: Seat status transitions (AVAILABLE → PENDING → RESERVED) must be atomic.
- **TypeScript integration**: The ORM layer must generate typed models consumed by our TypeScript codebase.
- **Startup-friendly**: Licensing cost, operational overhead, and community support all matter.

---

## Decision

Use **PostgreSQL** (latest stable) as the primary relational database, accessed through **Prisma ORM**.

## Why PostgreSQL Over MySQL

While MySQL is a reasonable alternative, PostgreSQL is the stronger choice for this platform for the following reasons:

### 1. Superior Concurrency for Reservation Workloads
Seat reservation is a classic concurrent-write problem. When two users attempt to reserve the same seat simultaneously, the database must enforce mutual exclusion. PostgreSQL's `SELECT FOR UPDATE` and advisory locks are more predictable and better documented than MySQL's equivalent, particularly under high-concurrency edge cases.

### 2. Stronger Standards Compliance
PostgreSQL adheres more strictly to the SQL standard. This matters because our Prisma-generated queries, raw SQL in migrations, and any future reporting queries will behave consistently and portably. MySQL's `GROUP BY` leniency and silent data truncation behaviours have historically caused subtle bugs in production systems.

### 3. JSONB for Flexible Metadata
PostgreSQL's `JSONB` type (binary JSON) is fully indexable and queryable. If we later need to store payment metadata, AXS callback payloads, or seat configuration as structured but schema-flexible data, `JSONB` handles this cleanly without a schema migration. MySQL's `JSON` type lacks the same indexing capability.

### 4. No Oracle Ownership Concern
MySQL is owned by Oracle Corporation. This has historically caused concern in the open-source community (it was the primary reason the MariaDB fork was created). PostgreSQL is governed by the independent PostgreSQL Global Development Group — no single corporate owner, no risk of licensing changes.

### 5. Cloud-Native Options at Zero Extra Cost
PostgreSQL is supported by every major cloud provider with managed services that offer connection pooling, automatic backups, point-in-time recovery, and read replicas — at no database license surcharge. Notable options relevant to our startup context:

| Provider | Service | Startup Benefit |
|----------|---------|----------------|
| Supabase | Supabase Postgres | Generous free tier; built-in auth and realtime |
| Neon | Neon Serverless Postgres | Serverless, scales to zero; ideal for early-stage |
| AWS | RDS PostgreSQL / Aurora | Enterprise-grade; easy migration path as we scale |
| GCP | Cloud SQL for PostgreSQL | Fully managed; integrates with GCP ecosystem |

## Why Not Oracle

Oracle Database is dismissed without extended evaluation:

- **Licensing cost**: Oracle charges per-CPU or per-named-user, with costs reaching tens of thousands of dollars annually. This is incompatible with a startup budget.
- **Operational complexity**: Oracle requires specialised DBAs and tooling that our TypeScript-first engineering team does not have.
- **Vendor lock-in**: Oracle's proprietary extensions (PL/SQL, Oracle-specific syntax) make migration costly if we ever need to move.
- **Prisma does not support Oracle**: Our chosen ORM has no Oracle connector, making it a non-starter for our stack.

## Why Prisma ORM

Database access is via **Prisma ORM** rather than a raw query builder or another ORM.

| Criterion | Prisma | TypeORM | Drizzle | Raw SQL (pg) |
|-----------|--------|---------|---------|-------------|
| Type-safe queries | Yes — auto-generated from schema | Partial — decorators, less ergonomic | Yes | No |
| Schema-as-code | Yes (`schema.prisma`) | Yes (entities) | Yes | No |
| Migration tooling | `prisma migrate` — declarative, version-controlled | Yes | Yes | Manual |
| TypeScript-first design | Yes | Partial | Yes | No |
| Learning curve | Low | Medium | Low-Medium | Low (but verbose) |

**Why Prisma wins**:
- The Prisma schema file is the single source of truth — from it, Prisma generates TypeScript types, the migration SQL, and the query client. No duplication between model definitions and TypeScript interfaces.
- `PrismaClient` integrates cleanly with Next.js API Route Handlers and supports connection pooling out of the box.
- `prisma migrate dev` gives a safe, reviewable migration workflow aligned with our Git-based development process.

## Concurrency Strategy for Seat Reservation

The most critical data-safety concern in this system is preventing two users from reserving the same seat. The strategy uses PostgreSQL transactions with row-level locking:

```
BEGIN;

  -- Lock the target seat row, blocking concurrent reservation attempts
  SELECT * FROM seats WHERE id = $seatId AND status = 'AVAILABLE'
  FOR UPDATE;

  -- If no row returned, seat is taken — rollback and return conflict error
  -- If row returned, proceed:

  UPDATE seats SET status = 'PENDING' WHERE id = $seatId;

  INSERT INTO reservations (seat_id, user_id, status) VALUES ($seatId, $userId, 'PENDING');

COMMIT;
```

This pattern guarantees that only one transaction can transition a seat from `AVAILABLE` to `PENDING` at a time. PostgreSQL's MVCC handles this correctly without application-level locking.

---

## Consequences

- Requires a running PostgreSQL instance — local Docker (`docker compose up postgres`) or a managed cloud service.
- Prisma Client must be regenerated (`prisma generate`) after any schema change.
- Connection pooling must be configured for production (Prisma Accelerate or PgBouncer) to avoid exhausting PostgreSQL's connection limit under load.
- The `SELECT FOR UPDATE` locking pattern must be used consistently in reservation mutation paths — this is a code discipline requirement documented in the implementation guide.
