# 3. Data Model

## 3.1 Purpose

This section describes the core domain entities of the Linkz Seat Reservation Platform, their relationships, and the lifecycle states that govern the reservation flow. This is a conceptual model — field-level definitions, schema, and constraints are covered in the Technical Specification Document (TSD).

---

## 3.2 Domain Entities

| Entity | Responsibility |
|--------|---------------|
| **User** | Represents an authenticated platform user. Identity is owned by Keycloak; this entity holds only the application-level profile linked by `keycloak_id`. |
| **Seat** | Represents one of the 3 reservable seats. Owns its own availability status and is the central resource that the reservation flow protects against concurrent access. |
| **Reservation** | Represents a user's intent to claim a seat. Bridges a User to a Seat and tracks the booking lifecycle from selection through to confirmation or cancellation. |
| **Payment** | Represents the AXS Checkout transaction associated with a reservation. Tracks the payment lifecycle and holds the AXS transaction reference for reconciliation and refund/void operations. |

---

## 3.3 Entity Relationships

```
┌─────────────┐          ┌──────────────────┐          ┌─────────────┐
│    User     │          │   Reservation    │          │    Seat     │
│             │ 1      * │                  │ *      1 │             │
│  (Keycloak  ├─────────►│  links User      ├─────────►│  owns its   │
│   identity) │          │  to Seat         │          │  status     │
└─────────────┘          └────────┬─────────┘          └─────────────┘
                                  │ 1
                                  │
                                  ▼ 1
                         ┌──────────────────┐
                         │     Payment      │
                         │                  │
                         │  AXS transaction │
                         │  reference       │
                         └──────────────────┘
```

- A **User** can have multiple Reservations over time, but only one active (PENDING or RESERVED) reservation at any point.
- A **Seat** can have multiple Reservations over time, but only one active reservation at any point — enforced at the database transaction level.
- A **Reservation** has exactly one **Payment**, created when the user proceeds to checkout.

## 3.4 Seat State Machine

A Seat is the central resource in the platform. Its status at any moment determines what actions are possible.

```
                    ┌───────────────┐
                    │   AVAILABLE   │◄──────────────────────┐
                    └──────┬────────┘                       │
                           │                                │
                    User selects seat                       │
                    + initiates payment                     │
                           │                                │
                           ▼                                │
                    ┌───────────────┐         Payment failed /
                    │    PENDING    │         cancelled / expired
                    └──────┬────────┘                       │
                           │                                │
                    AXS payment                             │
                    confirmed                               │
                           │                                │
                           ▼                                │
                    ┌───────────────┐                       │
                    │   RESERVED    │───── Cancellation ────┘
                    └───────────────┘      (refund / void)
```

| Transition | Trigger |
|-----------|---------|
| `AVAILABLE → PENDING` | User selects seat and payment is initiated |
| `PENDING → RESERVED` | AXS webhook confirms successful payment |
| `PENDING → AVAILABLE` | AXS payment fails, is cancelled, or `PENDING` reservation expires after **15 minutes** (aligned with AXS Checkout maximum payment page session duration) |
| `RESERVED → AVAILABLE` | Reservation is cancelled and refund/void is processed |

A Reservation and its Payment move through states together. The Payment state drives the Reservation outcome.

        │
        ▼

        │                                                │
        ▼                                                │
 Redirect to AXS Payment Page                           │
        │                                                │
   ┌────┴────┐                                           │

SUCCESS    FAILED / CANCELLED                            │
   │         │                                           │
   ▼         ▼                                           │

              │                                          │

## 3.6 Data Flow

### Read Path — Seat Availability

Seat availability is the most frequently read data and is served from cache wherever possible.

```
Incoming request (authenticated or guest)
        │
        ▼
Hazelcast Cache (embedded)
        │
   ┌────┴────┐
Cache HIT   Cache MISS
   │              │
   ▼              ▼
Return        PostgreSQL
cached     (seats table)
data              │
                  ▼
            Populate cache
                  │
                  ▼
            Return data
```

Cache entries are invalidated explicitly on every reservation state change — ensuring no user ever sees a stale seat status after a booking event. Full caching strategy is documented in [2.3. ADR - Caching](./2.3.%20ADR%20-%20Caching.md).

### Write Path — Reservation & Seat Status

All reservation writes go directly to PostgreSQL within a transaction, followed by immediate cache invalidation.

```
Reservation mutation (create / confirm / cancel)
        │
        ▼
PostgreSQL transaction
(seat row locked — prevents concurrent reservation)
        │
        ▼
Commit seat status + reservation state change
        │
        ▼
Evict Hazelcast cache entry
        │
        ▼
Next read re-populates cache from PostgreSQL
```

The concurrency strategy and locking approach are documented in [2.2. ADR - Database](./2.2.%20ADR%20-%20Database.md).
